Architecture Read

Application shell project.godot names the project Orbwalker, launches res://scenes/main_menu.tscn, uses a 1080x1920 portrait viewport, and autoloads RunState plus AudioManager.
Scene hosts scripts/scenes/ contains thin entry scripts for main menu, combat, shop, collection, and run summary, while feature folders hold models, views, controllers, services, and presenters.
Combat boundary scripts/combat/ has dedicated collaborators for state, model, view, controller, state machine, layout, HUD, VFX, resolve replay, debug console, outcome overlay, consumables, and chrome.
Board and content scripts/board/ owns match resolution and board interaction, while scripts/content/content_registry.gd centralizes equipment, consumables, mastery cards, relics, enemies, bosses, pricing, and validation snapshots.

The codebase has a clear model/view/controller flavor and has already extracted many high-risk combat responsibilities into presenter and service classes. The main remaining architectural pressure is coordination complexity: combat orchestration still has many responsibilities even after helper extraction.

Risks And Watch Items

Combat coordinator size: docs still flag combat orchestration as a future refactor candidate. The helper extractions reduce risk, but routing, replay order, HUD application, input, and turn flow remain a dense integration point.
Manual QA dependency: many recent UX changes are visual/timing-sensitive. The test suite helps protect deterministic logic, but mobile/tall-phone readability and interaction feel still need device review.
Asset pipeline gates: wiki pages note generated asset approval, legal/license review, and runtime integration gates. Production art integration should stay explicit and auditable.
Documentation drift: the HTML docs are extensive and useful. The cost is that feature pages, trackers, known issues, and architecture notes must be updated when ownership or behavior changes.
