## Overview

The project already covers the early prototype loop: board generation, drag movement, match resolution, combat, shop flow, boss rewards, run sequencing, content packs, and the player-facing HUD. (source: `todo.md`, `docs/test_plan.md`, `scripts/core/run_state.gd`)

## Details

### Board and combat

- 5x6 orb board with six orb types, deterministic generation, and no starting automatic matches. (source: `scripts/board/board_state.gd`, `todo.md`)
- Drag-based orb movement and board rendering are handled by the player-facing combat board path. The old board-debug scene/controller was removed in AR-08; focused editor-script probes now cover board resolver regression checks. (source: `scripts/board/board_view.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/debug/board_resolver_test_runner.gd`)
- Match resolution supports lines, L, T, gravity, refill, and cascades. (source: `docs/test_plan.md`, `scripts/board/board_match_resolver_v3.gd`)
- Combat resolves heart healing, armor gain, elemental damage, gold gain, block, and enemy death-before-intent behavior. (source: `scripts/combat/combat_state_machine.gd`, `docs/test_plan.md`)

### Run flow

- The run is a 3-level prototype with enemy, shop, boss, boss-reward, and advance steps. The normal player-facing boss reward step is presented inside the combat victory overlay, then continues to the post-boss shop. (source: `todo.md`, `scripts/core/run_state.gd`, `scripts/combat/combat_player_controller.gd`)
- Boss preview, boss relic reward, victory, and defeat flow are implemented in the run path. Boss relic reward selection now lives entirely in the combat victory overlay, and the old separate boss relic reward scene was removed in AR-08. Run summaries track total gold earned, monsters defeated, and bosses defeated separately. (source: `scripts/core/run_state.gd`, `scripts/combat/combat_player_controller.gd`)
- Run-flow entry, shop exits, and final-summary actions include local duplicate-transition guards for Start Run, player-shop Continue/Menu, and final-summary `Start New Run`/`Main Menu`. Start Run and final-summary actions now route through `RunState.flow_trace_change_scene(...)` and recover their buttons/status text on failed scene transitions; player-shop advance actions check failed `RunState` transition results before changing scenes, and player-shop Continue/Menu also unlock and show a failure status if the later traced scene-change call itself fails. The old boss reward and shop placeholder scenes were removed in AR-08. (source: `scripts/core/main_boot.gd`, `scripts/flow/shop_player.gd`, `scripts/flow/final_run_summary.gd`)
- Run-level balance logging is available through `RunState.run_log_snapshot()`, `RunState.run_log_export_json(...)`, `RunState.run_log_export_text()`, and `RunState.run_log_export_markdown()`. The log captures passive evidence for run start/end, fight start/end, turn results, shop open/actions/leave, and boss reward choice/skip. Shop log payloads include bounded item/relic offer snapshots with content ids, display names, types, prices, affordability, sold-out state, booster presence, reroll state, pending booster options, selected/purchased/granted action details, and shop before/after snapshots for actions and leaving. The main menu exposes a persisted `Generate Log` toggle that defaults off; when enabled, finalizing a run writes JSON, Markdown, and text reports under project-root `logs/`. `RunState.run_log_last_export_snapshot()` exposes the latest paths/errors. Text and Markdown formatting live in `scripts/core/run_log_reporter.gd`. (source: `scripts/core/run_state.gd`, `scripts/core/run_log_reporter.gd`, `scripts/core/main_boot.gd`, `scripts/combat/combat_player_controller.gd`, `.gitignore`)
- Milestone 10 has a temporary prototype balance lever surface and a validated playtest baseline for Milestone 11. M10-04 currently uses fixed fight base rewards instead of random early gold boosts: new runs start with `0` gold, Gold orb spawn/shop/reroll multipliers stay neutral at `1.0`, and level 1/2/3 fights grant base rewards of `10/12/14` gold through `RunState.mark_fight_victory()` after matched-gold gains resolve. The post-match victory popup displays total fight gold as base plus matched gold. The first level-1 shop guarantees affordable `Shortsword` damage access at the 10-gold floor, and the shop pool guarantees one booster when boosters are available while keeping equipment dominant and consumables rare. M10-05 adds level/type-scoped temporary survivability multipliers while keeping global enemy HP/damage multipliers neutral. Dungeon 1 is a forgiving damage check with frequent enemy block/defend intents: level 1 normal HP/damage `0.50/0.50`, level 1 boss `0.60/0.65`. Dungeon 2 is the first defense check with stronger/more frequent attacks and lighter block: level 2 normal `0.90/1.0`, level 2 boss `1.0/1.10`. Dungeon 3 is now a late damage check after high-effort playtest feedback: level 3 normal HP/damage `2.2/1.20`, level 3 boss `2.60/1.30`. M10-07 closeout logs show the current layer can produce both late-run defeats and a full victory, so remaining tuning should wait until after Milestone 11 meta progression. These values are temporary playtest scaffolding, not final balance. (source: `scripts/core/run_state.gd`, `scripts/combat/combat_turn_logger.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/shop/shop_service.gd`, `docs/milestone_10_balance_tasks.md`, `docs/test_plan.md`)

### Shop and progression

- The shop offers 3 random item slots and 1 relic offer per dungeon level, plus reroll, buying, selling, and booster selection. Generated item stock and booster options exclude equipment that is already equipped, and relic offers exclude relics already owned by the player, including cached per-level offers that become stale after ownership changes. Pending booster choices do not block the shared player HUD; full-slot rewards keep the booster choices open so the player can sell equipped gear or consumables from the loadout bubble or press Skip. (source: `docs/game_design_document.md`, `scripts/shop/shop_service.gd`, `scripts/flow/shop_player.gd`)
- Player progression tracks 5 equipment slots, 3 consumable slots, relic ownership, and 6 mastery tracks capped at 5. (source: `scripts/run/player_progression_state.gd`, `scripts/run/player_progression_service.gd`)
- Milestone 11 adds persistent equipment achievement progression through a default `PlayerProfileState` profile that contains `MetaProfileState`. Run Score starts at `0` each run, increases from non-sell gold sources, excludes sell/refund gold, and banks into persistent Total Score when the run ends. The player profile stores profile identity plus Total Score, unlocked equipment ids, and recent equipment unlock toast payloads separately from temporary `RunState`; the previous flat meta-profile save is migrated into the default profile. (source: `scripts/core/run_state.gd`, `scripts/run/player_profile_state.gd`, `scripts/run/meta_profile_state.gd`)
- Equipment progression currently covers 5 families with max rarity Rare: Shortsword, Buckler, Coin Purse, Healing Charm, and Leather Gloves. Common variants are unlocked by default; winning with a Common equipped unlocks that family's Uncommon, winning with an Uncommon equipped unlocks that family's Rare, and the Collection screen can claim adjacent unlocks for Total Score costs of `100` and `300`. Locked variants cannot roll in shops or booster choices, unlocked lower rarities can still roll, and equipping two variants from the same family is rejected. (source: `scripts/content/content_registry.gd`, `scripts/core/run_state.gd`, `scripts/run/player_progression_service.gd`, `scripts/shop/shop_service.gd`, `scripts/flow/collection.gd`)

### UI and presentation

- A lightweight `AudioManager` now plays exported MIDI-source WAV music for menu, combat, and shop when the assets exist, with code-generated fallback loops and short generated SFX tones. Exported music is peak-normalized by the render script and played at `-12 dB`; shared WAV loading, source-frame counting, and loop configuration live in `AudioStreamLoader`, which can decode source signed 16-bit WAV into memory or configure Godot imported streams. Android/template builds now try non-imported raw WAV byte copies (`res://resources/audio/raw_music/{menu,combat,shop}.wav.bin`) first so exported APK playback keeps full uploaded WAV data, then fall back to `res://resources/audio/music/*.wav`, and finally generated loops if both fail. Menu, combat, and shop scenes request their own looping music contexts, while SFX hooks cover menu start, combat drag swaps, combat match/combo/result/victory/defeat events, and shop purchase/reroll/sell/booster success or failure feedback. `AudioManagerResolver` owns the shared dynamic `/root/AudioManager` lookup and lazy already-open-editor fallback used by menu, combat, and shop controllers. Verbose music diagnostics are opt-in through `debug/audio_diagnostics_enabled`. (source: `scripts/core/audio_manager.gd`, `scripts/core/audio_stream_loader.gd`, `scripts/core/audio_manager_resolver.gd`, `resources/audio/music/`, `resources/audio/raw_music/`, `project.godot`, `scripts/core/main_boot.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/flow/shop_player.gd`)

- Combat board now shows one floating combo counter that updates from `COMBO x1` upward through each visible resolve/cascade step instead of stacking multiple match-group popups. Drag release clears leftover swap overlays, assigns `BoardView` to a pre-resolve visual clone, runs the resolver against a separate simulation clone, replays clear, gravity, and refill into the visual board, then commits the final simulation board after presentation. Presentation is trigger-based and ordered: within each resolve pass, match groups are sorted by top row then left column, and each group flashes, clears, commits visually, and advances the shared `COMBO xN` popup plus matching Elemental Mastery preview before the next same-pass group starts.
