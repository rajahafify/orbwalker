(source: docs/itch_readiness_tasks.html, scripts/combat/combat_controller.gd)
  • AR-26 signal capture currently deep-copies _run_summary in RunState._capture_run_signal_state() and compares full dictionaries in _emit_run_summary_changed_if_needed(...). This preserves correct change detection and is not a current blocker, but it is unnecessary work for ordinary gold/step mutations after a run summary exists. A future cleanup could replace the deep-copy/equality path with a _run_summary_generation counter incremented when summaries are finalized, cleared, or restored, while preserving current signal payloads. (source: scripts/core/run_state.gd, docs/architecture_review_tasks.html)
  • The combat scene controller remains large after the AR helper extractions: the legacy root controller was 2432 lines on the AR-18 branch before it was renamed to scripts/scenes/combat.gd. The god-object risk is reduced because debug console, turn-log formatting, layout, VFX, drag input, placeholder textures, visual chrome, resolve presentation, outcome overlay presentation, and HUD snapshot building are extracted, but combat flow routing, scene-node HUD application, and turn orchestration remain in the controller and should be treated as future coordinator-boundary refactor candidates. (source: docs/architecture_review_tasks.html, scripts/scenes/combat.gd)
  • Shared HUD API drift risk is narrowed after the scene-review cleanup because named HUD layout presets now live in PlayerLoadoutHud; future screens that need alternate HUD geometry should add shared presets there instead of screen-local internal HUD constants. (source: docs/architecture_review_tasks.html, scripts/ui/player_loadout_hud.gd, scripts/scenes/shop.gd)
  • Temporary diagnostics are retained intentionally for Milestone 10/M12 QA: FlowTrace logs exposed through RunState and stored by SceneRouter, combat ResolveTrace logs, RunState.flow_trace_debug_snapshot(), and the feature-flagged AR-01 combat result-envelope probe. FlowTrace route state is capped at 50 retained routes, so the diagnostic no longer grows unbounded during long sessions. These diagnostics should be retired, feature-flagged further, or moved into a narrower debug harness if they outlive balance/build-readiness QA usefulness. (source: docs/architecture_review_tasks.html, docs/test_plan.html, scripts/core/run_state.gd, scripts/core/scene_router.gd, scripts/scenes/combat.gd, scripts/debug/ar01_combat_result_probe.gd)
  • AR-01 baseline capture found repeated unsourced Godot GDScript::reload: Integer division. Decimal part will be discarded. warnings after scene smokes. Later AR batches repeatedly reached no-session-error get_godot_errors gates, but if the warning reappears it should still be localized before treating error output as a clean regression signal. (source: docs/test_plan.html)

Clean release-candidate error gating now lives in the itch readiness tracker. (source: docs/itch_readiness_tasks.html, scripts/combat/combat_controller.gd)
  • Transferred to ITCH-06: Android CLI export on this Windows checkout can hang after writing a valid Orbwalker.apk; public export hygiene and workaround validation now live in the itch packaging slice. (source: wiki/setup.html, docs/test_plan.html, docs/itch_readiness_tasks.html)
  • Balance tuning is tracked through the Milestone 10 task tracker. The M10-01 inventory confirmed the active tuning owners before value changes: shop pricing and much prototype content live in dictionary-backed ContentRegistry, while current runtime encounter selection and enemy/boss stats are still owned by RunState. ContentRegistry still carries older enemy rows for contract/content coverage, so active enemy HP/intent tuning should use RunState unless ownership is deliberately migrated later. (source: docs/milestone_10_balance_tasks.html, scripts/content/content_registry.gd, scripts/core/run_state.gd, todo.html, docs/game_design_document.html)
  • M10-02 captured 3 human-played untuned baseline logs before tuning. The evidence showed mixed blockers: first-shop affordability could be weak, low-effectiveness play could die before shop access, and stronger runs still hit level 2-3 survivability pressure. M10-03 through M10-06 added temporary balance levers, fixed fight base rewards (10/12/14), first-shop Shortsword affordability, booster presence, equipment-heavy shop stock, rare consumables, and level/type-scoped survivability identities. M10-07 closeout compared tuned logs against the untuned baseline and found no Milestone 10 blocker: run_1777968781_770133_2026-05-05t16_13_01 died in Dungeon 3, run_1777969048_434533_2026-05-05t16_17_28 won a full run, and run_1777973747_694854_2026-05-05t17_35_47 reached L3 enemy 2 after first-shop affordability, booster buys, equipment buys, and two boss relic rewards. Full-slot equipment friction is visible in late Dungeon 3 shops, but it is a post-Milestone-11 UX/economy tuning question unless player testing says it blocks the run. Deeper combat/economy rebalance is deferred until after meta progression. These values are temporary M10 playtest scaffolding, not final balance. (source: docs/milestone_10_balance_tasks.html, docs/test_plan.html, scripts/core/run_state.gd, scripts/combat/combat_turn_log_presenter.gd, scripts/shop/shop_service.gd, logs/run_1777973747_694854_2026-05-05t17_35_47.json)
