## Overview

The architecture is organized around state-owned gameplay logic and scene-driven presentation. `RunState` remains the public autoload facade for run flow, run gold, dungeon level, boss rewards, encounter selection, routing APIs, Run Log APIs, profile/meta APIs, prototype balance APIs, and additive state-change signals. `SceneRouter` owns FlowTrace/scene-transition machinery, `RunLogger` owns run-log storage/export state, `ProfileRepository` owns profile file load/save and legacy migration, and `BalanceManager` owns prototype balance defaults, lever state, ProjectSettings sync, and encounter scaling behind that facade. `BoardModel` owns the 5x6 grid. `CombatStateMachine` resolves turns. `ShopService` owns shop actions. `PlayerProgressionState` and `PlayerProgressionService` own equipment, consumables, relics, and mastery transitions. (source: `scripts/core/run_state.gd`, `scripts/core/scene_router.gd`, `scripts/core/run_logger.gd`, `scripts/core/profile_repository.gd`, `scripts/core/balance_manager.gd`, `scripts/board/board_model.gd`, `scripts/combat/combat_state_machine.gd`, `scripts/shop/shop_service.gd`, `scripts/run/player_progression_state.gd`, `scripts/run/player_progression_service.gd`)

## Details

### Run flow

- `RunState` tracks dungeon level, current step, current encounter, run gold, total gold earned, and the boss-reward sequence while exposing compatibility wrappers for routing and Run Log APIs. It also emits additive dictionary-payload signals for committed gold, run step, run active/victory/summary, profile/meta, and run-summary changes; existing controllers still use the polling/snapshot APIs unless deliberately migrated later. `SceneRouter` chooses and attaches the next scene, owns FlowTrace route records, retains only the latest 50 route records while preserving the active route, and uses a transition-generation guard to avoid restoring stale snapshots after newer run/transition mutations. `RunLogger` owns event serials, event storage, run metadata, snapshot/restore data, and end-of-run export bookkeeping; `RunLogReporter` still formats readable reports. (source: `scripts/core/run_state.gd`, `scripts/core/scene_router.gd`, `scripts/core/run_logger.gd`, `scripts/core/run_log_reporter.gd`)
- `ProfileRepository` owns profile and legacy meta-profile `ConfigFile` persistence, including `user://matchatro_profile.cfg`, legacy `user://matchatro_meta_profile.cfg` migration, save warnings, and last-I/O probe metadata. `RunState` still owns when profile/meta state is created, reset, synced with default equipment unlocks, mutated for score/unlocks, and exposed through compatibility snapshots. (source: `scripts/core/profile_repository.gd`, `scripts/core/run_state.gd`)
- `BalanceManager` owns prototype balance defaults, current lever state, lever normalization, ProjectSettings sync under `matchatro/prototype_balance/`, starting-gold lookup, fight reward lookup, ContentRegistry lever propagation, and encounter HP/damage scaling. `RunState` still exposes the public prototype balance methods, owns active encounter dictionaries, and decides when scaling is applied during fight assignment. (source: `scripts/core/balance_manager.gd`, `scripts/core/run_state.gd`, `scripts/content/content_registry.gd`)
- The level structure is a 3-level prototype run with repeated enemy, shop, boss reward, and advance steps. (source: `todo.md`, `scripts/core/run_state.gd`)

### Board and combat

- `BoardModel` is a pure data model with deterministic generation, bounds checks, swaps, and match-avoidant starting boards. (source: `scripts/board/board_model.gd`)
- `BoardView` is a render-and-animation surface. It draws the grid, selected cells, drag path, flashes, and glow overlays, owns board presentation model binding, drag visual reset/update helpers, hover lookup, and board input filtering, but does not own board rules. (source: `scripts/board/board_view.gd`)
- `CombatStateMachine` resolves the turn order, applies player effects in the documented sequence, handles block and armor, and records a structured combat log. External callers that need to reset phase now use `reset_to_intent_preview()` instead of directly writing the public `phase` field; existing read access remains for HUD/debug checks. (source: `scripts/combat/combat_state_machine.gd`, `scripts/scenes/combat.gd`)
- Combat now follows the project-local MVC host pattern. `scripts/scenes/combat.gd` is the thin scene host, `CombatModel` owns small scene-local mutable state, `CombatView` owns node binding/snapshot access, and `CombatController` owns combat scene orchestration, input phase authority, resolver simulation, combat math handoff, RunState outcome routing, audio hooks, `/skip` route/state reset, privileged debug callbacks, HUD snapshot application and `PlayerLoadoutHud` payload dispatch, timer runtime text/fill/color math, placeholder fallback decisions and scene-node assignment, VFX timing decisions, and scene transitions. Its `_trace_and_change_scene_to_target(...)` helper owns the duplicated combat outcome trace/change-scene glue for standard Continue, boss reward claim, and boss reward skip without taking over `RunState` route semantics or `CombatOutcomeOverlay` presentation. `CombatHudPresenter` owns side-effect-free combat HUD snapshot dictionary construction for top HUD, enemy stage, timer/tempo row, player strip, and debug overlay data. `CombatPlaceholderTextures` owns only the code-generated timer, intent, enemy portrait, and hero portrait placeholder texture builders. `CombatChromeStyler` owns code-built combat chrome/style construction, including shared combat frame styleboxes, progress-bar styles, label font/color overrides, timer-track and timer-label readability styling, button chrome, board/outcome panel chrome, stat-chip chrome, debug overlay font sizing, shared player-HUD chrome dispatch, and debug zone-guide chrome. `BoardController` owns guarded `BoardView.gui_input`/`mouse_exited` wiring, board-local mouse/touch event parsing, drag-result callback dispatch, board-local hovered-orb-id translation, active drag state, touch-index tracking, selected orb/current cell/path tracking, adjacent-cell swap bookkeeping, move-timer countdown state, drag visual reset/abort, match-glow refresh, visual replay mutation APIs for clear/gravity/refill commits, board input-enable forwarding, and final board-model commit while preserving `BoardView.gui_input` local coordinates. `CombatLayoutPresenter` owns combat scene geometry and responsive design-space positioning, including design-root scaling, runtime zone rects, enemy/strip/board/player panel positioning, loadout rail positioning, debug overlay anchors, and outcome overlay layout sync. `CombatVfxPresenter` owns transient combat VFX drawing mechanics: VFX layer binding, texture VFX spawning, replay impacts, mastery beam source lookup, global-to-layer coordinate conversion, beam sizing/rotation/z-index, and fade cleanup. `CombatDebugConsole` currently owns debug command parsing/dispatch, command help/error text, log-level state, and combat log rendering. Future AR-27 should split log rendering into a reusable `ConsoleLogger` scene that listens to shared console-log events from any scene, while combat command dispatch stays combat-specific unless a later command-router task is accepted. `CombatTurnLogPresenter` owns normal/detailed turn-log text, state snapshot formatting helpers, intent text, and reusable outcome/summary strings. `CombatResolvePresenter` owns only the board-space resolve replay presentation boundary: sorted match presentation order, match flash waits, clear/gravity/refill animation timing, board-controller delegated visual commit steps, clear bursts, combo popup lifecycle, and `combat_speed` waits. `CombatOutcomeOverlay` owns only the combat outcome overlay presentation boundary: standard victory/defeat card state, boss reward card controls, scrim layering, card layout/content, and text wrapping. (source: `scripts/scenes/combat.gd`, `scripts/combat/combat_controller.gd`, `scripts/combat/combat_model.gd`, `scripts/combat/combat_view.gd`, `scripts/combat/combat_hud_presenter.gd`, `scripts/combat/combat_placeholder_textures.gd`, `scripts/combat/combat_chrome_styler.gd`, `scripts/board/board_controller.gd`, `scripts/combat/combat_layout_presenter.gd`, `scripts/combat/combat_vfx_presenter.gd`, `scripts/combat/combat_debug_console.gd`, `scripts/combat/combat_turn_log_presenter.gd`, `scripts/combat/combat_resolve_presenter.gd`, `scripts/combat/combat_outcome_overlay.gd`)
- Architecture-review refactors extracted obvious combat leaf responsibilities and then moved combat to the local MVC shape. The current accepted boundary is structural rather than final: `CombatController` is still large, `CombatView` is still conservative, and follow-up cleanup should target clear ownership names and real seams, especially reusable console logging, view-ready HUD presentation, combat layout/VFX presenter naming, and log presentation. (source: `docs/architecture_review_tasks.md`, `scripts/scenes/combat.gd`, `scripts/combat/combat_controller.gd`)

### Shop and progression

- `ShopService` remains the economy and shop-state authority that opens shops, rerolls offers, buys items, sells equipped items, and resolves booster picks through `RunState` and the progression service. (source: `scripts/shop/shop_service.gd`)
- Shop runtime scene ownership now follows a local MVC split: `ShopModel` owns selected-slot/status/transition state plus enriched shop snapshot projection, `ShopView` owns the shop UI surface and shared `PlayerLoadoutHud` binding/layout/rendering, `ShopController` owns shop actions, RunState transition calls, scene-change rollback callbacks, FlowTrace marks, and shop music/SFX hooks, and `scripts/scenes/shop.gd` is a thin host that wires scene nodes to model/view/controller. (source: `scripts/shop/shop_model.gd`, `scripts/shop/shop_view.gd`, `scripts/shop/shop_controller.gd`, `scripts/scenes/shop.gd`)
- Main-menu, Collection, and final run-summary screens now follow the same local MVC ownership pattern as shop. `MainMenuModel`, `CollectionModel`, and `RunSummaryModel` own screen-local state snapshots and formatting helpers; corresponding `*View` scripts own node binding, screen rendering/layout/chrome, status labels, and toast/profile presentation; corresponding `*Controller` scripts own lifecycle entry, action routing, `RunState`/FlowTrace transition calls, transition locks, and audio command routing. `scripts/scenes/main_menu.gd`, `scripts/scenes/collection.gd`, and `scripts/scenes/run_summary.gd` are thin hosts that wire nodes and forward signals to controllers. (source: `scripts/main_menu/main_menu_model.gd`, `scripts/main_menu/main_menu_view.gd`, `scripts/main_menu/main_menu_controller.gd`, `scripts/collection/collection_model.gd`, `scripts/collection/collection_view.gd`, `scripts/collection/collection_controller.gd`, `scripts/run_summary/run_summary_model.gd`, `scripts/run_summary/run_summary_view.gd`, `scripts/run_summary/run_summary_controller.gd`, `scripts/scenes/main_menu.gd`, `scripts/scenes/collection.gd`, `scripts/scenes/run_summary.gd`)
- `PlayerProgressionState` stores equipped items, consumables, relics, and mastery levels. `PlayerProgressionService` performs the legal transitions and rebuilds active effects. (source: `scripts/run/player_progression_state.gd`, `scripts/run/player_progression_service.gd`)

### Content and validation

- `ContentRegistry` currently stores content as nested dictionaries and validates required fields, effect hooks, entry types, and `next_tier_item_id` references. It also builds shop pools and pricing snapshots, clamps shop/reroll pricing multipliers to a positive prototype floor, and exposes the reroll cost ceiling used by `ShopService`. `content_contract_snapshot()` records the current collection fields, validation owner, shop pool/pricing owner, and future migration boundary. Single-item getters and list APIs return duplicated dictionaries so callers cannot mutate the registry index by accident. (source: `scripts/content/content_registry.gd`, `scripts/shop/shop_service.gd`)
- Dictionary-backed `ContentRegistry` content is the prototype source of truth for this phase. The older Resource-based architecture direction is now documented as future migration work that should stay behind the `ContentRegistry` read API instead of changing combat, shop, progression, HUD, or debug callers directly. (source: `docs/system_architecture.md`, `scripts/content/content_registry.gd`)

### Visual routing

- `visual_registry.gd` centralizes first-pass texture lookup and fallback behavior for combat and shop presentation. (source: `scripts/ui/visual_registry.gd`)

### Scene structure

- The live `scenes/` folder keeps player-facing root screens flat: `main_menu.tscn`, `combat.tscn`, `shop.tscn`, `run_summary.tscn`, and `collection.tscn`. Reusable scene components remain under `scenes/ui/`, including `board.tscn` and `player_hud.tscn`. Root scene scripts mirror those screens under `scripts/scenes/`; reusable systems remain in domain folders such as `scripts/board/`, `scripts/combat/`, `scripts/shop/`, and `scripts/ui/`. (source: repository layout, `project.godot`)
- Shared UI composition follows the same ownership rule as shared runtime helpers: combat and shop may mount shared components in different screen zones, but should not redefine shared component internals with screen-local geometry constants. `PlayerLoadoutHud` now owns both the default shared HUD geometry and the named shop portrait layout preset, while `CombatLayoutPresenter` owns combat's runtime section override and `ShopView` owns the shop screen mount/placement. (source: `scripts/shop/shop_view.gd`, `scripts/combat/combat_layout_presenter.gd`, `scripts/ui/player_loadout_hud.gd`)
