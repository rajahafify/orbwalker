Overview

The project already covers the early prototype loop: board generation, drag movement, match resolution, combat, shop flow, boss rewards, run sequencing, content packs, and the player-facing HUD. (source: todo.html, docs/test_plan.html, scripts/core/run_state.gd)

Details

Board and combat

  • 5x6 orb board with six orb types, deterministic generation, and no starting automatic matches. (source: scripts/board/board_model.gd, todo.html)
  • Drag-based orb movement and board rendering are handled by the player-facing combat board path. The old board-debug scene/controller was removed in AR-08; focused editor-script probes now cover board resolver regression checks. The combat scene host delegates runtime orchestration to CombatController, while BoardController still owns board-local BoardView.gui_input handling. (source: scripts/board/board_view.gd, scripts/board/board_controller.gd, scripts/scenes/combat.gd, scripts/combat/combat_controller.gd, scripts/tests/integrations/board_resolver_test.gd)
  • Match resolution supports lines, L, T, gravity, refill, and cascades. (source: docs/test_plan.html, scripts/board/board_match_resolver_service.gd)
  • Combat resolves heart healing, armor gain, elemental damage, gold gain, block, and enemy death-before-intent behavior. (source: scripts/combat/combat_state_machine.gd, docs/test_plan.html)

Run flow

  • The main menu includes a dedicated TUTORIAL action that starts a tutorial run and routes through FlowTrace into the normal res://scenes/combat.tscn surface. Tutorial mode uses the real combat UI, normal run progression, normal shop UI, deterministic combat board seeds per turn, and deterministic shop RNG per shop/reroll so first-time-player practice is repeatable without a separate mock tutorial screen. The first tutorial combat displays in-combat dialogue and board coachmarks for the predetermined Gold-to-Fire match, enemy attack intent, Shield/Armor blocking, Heart healing, and a multi-combo finisher. The first shop tutorial forces buying Iron Shortsword, rerolling, and continuing through normal shop controls, then the next combat explains the bought Shortsword and Mastery in separate animated highlight steps before showing the end-of-tutorial Continue/Main Menu modal. (source: scenes/main_menu.tscn, scripts/scenes/main_menu.gd, scripts/main_menu/main_menu_controller.gd, scripts/main_menu/main_menu_view.gd, scripts/core/run_state.gd, scripts/combat/combat_controller.gd, scripts/combat/combat_view.gd, scripts/board/board_controller.gd, scripts/board/board_view.gd, scripts/shop/shop_model.gd, scripts/shop/shop_controller.gd, scripts/shop/shop_view.gd, scripts/shop/shop_service.gd)
  • The run is a 3-level prototype with enemy, shop, boss, boss-reward, and advance steps. The normal player-facing boss reward step is presented inside the combat victory overlay, then continues to the post-boss shop. (source: todo.html, scripts/core/run_state.gd, scripts/scenes/combat.gd)
  • Boss preview, boss relic reward, victory, and defeat flow are implemented in the run path. Boss relic reward selection now lives entirely in the combat victory overlay, and the old separate boss relic reward scene was removed in AR-08. Run summaries track total gold earned, monsters defeated, and bosses defeated separately. (source: scripts/core/run_state.gd, scripts/scenes/combat.gd)
  • Run-flow entry, shop exits, and final-summary actions include local duplicate-transition guards for Start Run, player-shop Continue/Menu, and final-summary Start New Run/Main Menu. Start Run and final-summary actions now route through RunState.flow_trace_change_scene(...) and recover their buttons/status text on failed scene transitions; player-shop advance actions check failed RunState transition results before changing scenes, and player-shop Continue/Menu also unlock and show a failure status if the later traced scene-change call itself fails. The old boss reward and shop placeholder scenes were removed in AR-08. (source: scripts/scenes/main_menu.gd, scripts/scenes/shop.gd, scripts/scenes/run_summary.gd)
  • Run-level balance logging is available through RunState.run_log_snapshot(), RunState.run_log_export_json(...), RunState.run_log_export_text(), and RunState.run_log_export_html(). The log captures passive evidence for run start/end, fight start/end, turn results, shop open/actions/leave, and boss reward choice/skip. Shop log payloads include bounded item/relic offer snapshots with content ids, display names, types, prices, affordability, sold-out state, treasure chest presence, reroll state, pending treasure chest options, selected/purchased/granted action details, and shop before/after snapshots for actions and leaving. The main menu exposes a persisted Generate Log toggle that defaults off; when enabled, finalizing a run writes JSON, HTML, and text reports under project-root logs/. RunState.run_log_last_export_snapshot() exposes the latest paths/errors. Text and HTML formatting live in scripts/core/run_log_reporter.gd. (source: scripts/core/run_state.gd, scripts/core/run_log_reporter.gd, scripts/scenes/main_menu.gd, scripts/scenes/combat.gd, .gitignore)
  • Milestone 10 has a temporary prototype balance lever surface and a validated playtest baseline for Milestone 11. M10-04 currently uses fixed fight base rewards instead of random early gold boosts: new runs start with 0 gold, Gold orb spawn/shop/reroll multipliers stay neutral at 1.0, and level 1/2/3 fights grant base rewards of 10/12/14 gold through RunState.mark_fight_victory() after matched-gold gains resolve. The post-match victory popup displays total fight gold as base plus matched gold. The first level-1 shop guarantees affordable Shortsword damage access at the 10-gold floor, and the shop pool guarantees one treasure chest when treasure chests are available while keeping equipment dominant and consumables rare. M10-05 adds level/type-scoped temporary survivability multipliers while keeping global enemy HP/damage multipliers neutral. Dungeon 1 is a forgiving damage check with frequent enemy block/defend intents: level 1 normal HP/damage 0.50/0.50, level 1 boss 0.60/0.65. Dungeon 2 is the first defense check with stronger/more frequent attacks and lighter block: level 2 normal 0.90/1.0, level 2 boss 1.0/1.10. Dungeon 3 is now a late damage check after high-effort playtest feedback: level 3 normal HP/damage 2.2/1.20, level 3 boss 2.60/1.30. M10-07 closeout logs show the current layer can produce both late-run defeats and a full victory, so remaining tuning should wait until after Milestone 11 meta progression. These values are temporary playtest scaffolding, not final balance. (source: scripts/core/run_state.gd, scripts/combat/combat_turn_log_presenter.gd, scripts/scenes/combat.gd, scripts/shop/shop_service.gd, docs/milestone_10_balance_tasks.html, docs/test_plan.html)

Shop and progression

  • The shop offers 3 random item slots and 1 relic offer per dungeon level, plus reroll, buying, selling, and treasure chest selection. Generated item stock and treasure chest options exclude equipment that is already equipped, and relic offers exclude relics already owned by the player, including cached per-level offers that become stale after ownership changes. Pending treasure chest choices do not block the shared player HUD; full-slot rewards keep the treasure chest choices open so the player can sell equipped gear or consumables from the loadout bubble or press Skip. Runtime APIs now use the treasure_chest type/key names. (source: docs/game_design_document.html, scripts/shop/shop_service.gd, scripts/scenes/shop.gd)
  • Shop runtime presentation now follows a local MVC split: ShopModel projects enriched offer state for rendering (affordable, sold_out, disabled, reroll_enabled) and tracks local selection/status/transition lock; ShopView owns shop UI build/layout/rendering plus shared PlayerLoadoutHud binding/signals; and ShopController owns shop actions, RunState transition calls/rollback callbacks, FlowTrace markers, and shop audio hooks while preserving existing RunState and ShopService APIs. (source: scripts/shop/shop_model.gd, scripts/shop/shop_view.gd, scripts/shop/shop_controller.gd, scripts/scenes/shop.gd, scripts/shop/shop_service.gd)
  • Non-combat player-facing screens now follow the same local MVC ownership split. Main menu (MainMenuModel/MainMenuView/MainMenuController), Collection (CollectionModel/CollectionView/CollectionController), and final run summary (RunSummaryModel/RunSummaryView/RunSummaryController) keep screen-local state/format helpers in model, node rendering/layout/chrome/status/toast/profile surface in view, and RunState routing/FlowTrace/action locks/audio commands in controller while preserving the same scene paths and visible behavior. (source: scripts/main_menu/main_menu_model.gd, scripts/main_menu/main_menu_view.gd, scripts/main_menu/main_menu_controller.gd, scripts/collection/collection_model.gd, scripts/collection/collection_view.gd, scripts/collection/collection_controller.gd, scripts/run_summary/run_summary_model.gd, scripts/run_summary/run_summary_view.gd, scripts/run_summary/run_summary_controller.gd, scripts/scenes/main_menu.gd, scripts/scenes/collection.gd, scripts/scenes/run_summary.gd)
  • Combat now follows the same local MVC host shape while preserving domain ownership: scripts/scenes/combat.gd is the thin host, CombatView owns combat rendering, status/turn-summary/debug/outcome UI boundaries, shared PlayerLoadoutHud binding/render/control wrappers, shared HUD node-map assembly, sole CombatLayoutPresenter wiring/application, VFX layer visibility, and enemy/player VFX target geometry helpers, CombatModel owns only scene-local mutable bookkeeping such as input phase, transition locks, staged HUD replay values and mutation helpers, mastery preview totals/release bookkeeping, and resolve trace state, and CombatController keeps combat flow orchestration, domain state collaborators, RunState/FlowTrace routing, debug callbacks, consumable-use eligibility/status/audio consequences, VFX timing decisions, and replay timing decisions. The latest controller cleanup also removed leftover wrapper/delegate helpers, so normal timer/intent hover and replay VFX paths now call CombatView and CombatVfxPresenter directly while preserving existing replay order and combat_speed timing. CombatConsumableService applies combat-scoped consumable effect dictionaries through injected board conversion callbacks. (source: scripts/scenes/combat.gd, scripts/combat/combat_model.gd, scripts/combat/combat_view.gd, scripts/combat/combat_controller.gd, scripts/combat/combat_vfx_presenter.gd, scripts/combat/combat_consumable_service.gd)
  • Player progression tracks 5 equipment slots, 3 consumable slots, relic ownership, and 6 mastery tracks capped at 5. (source: scripts/run/player_progression_state.gd, scripts/run/player_progression_service.gd)
  • Milestone 11 adds persistent equipment achievement progression through a default PlayerProfileState profile that contains MetaProfileState. Run Score starts at 0 each run, increases from non-sell gold sources, excludes sell/refund gold, and banks into persistent Total Score when the run ends. The player profile stores profile identity plus Total Score, unlocked equipment ids, and recent equipment unlock toast payloads separately from temporary RunState; the previous flat meta-profile save is migrated into the default profile. (source: scripts/core/run_state.gd, scripts/run/player_profile_state.gd, scripts/run/meta_profile_state.gd)
  • Equipment progression currently covers 5 families with max rarity Rare: Shortsword, Buckler, Coin Purse, Healing Charm, and Leather Gloves. Common variants are unlocked by default; winning with a Common equipped unlocks that family's Uncommon, winning with an Uncommon equipped unlocks that family's Rare, and the Collection screen can claim adjacent unlocks for Total Score costs of 100 and 300. Locked variants cannot roll in shops or treasure chest choices, unlocked lower rarities can still roll, and equipping two variants from the same family is rejected. (source: scripts/content/content_registry.gd, scripts/core/run_state.gd, scripts/run/player_progression_service.gd, scripts/shop/shop_service.gd, scripts/scenes/collection.gd)

Content pack

  • The current content registry includes equipment, mastery cards, consumables, relics, enemies, and bosses. (source: scripts/content/content_registry.gd)
  • After the Milestone 11 progression slice, the active equipment pack is 15 equipment variants: 5 Common, 5 Uncommon, and 5 Rare across 5 families. The registry still includes 6 mastery cards, 6 consumables, 5 relics, 3 enemies, and 3 bosses. (source: docs/test_plan.html, scripts/content/content_registry.gd)

UI and presentation

  • A lightweight AudioManager now plays exported MIDI-source WAV music for menu, combat, and shop when the assets exist, with code-generated fallback loops and short generated SFX tones. Exported music is peak-normalized by the render script and played at -12 dB; shared WAV loading, source-frame counting, and loop configuration live in AudioStreamLoader, which can decode source signed 16-bit WAV into memory or configure Godot imported streams. Android/template builds now try non-imported raw WAV byte copies (res://resources/audio/raw_music/{menu,combat,shop}.wav.bin) first so exported APK playback keeps full uploaded WAV data, then fall back to res://resources/audio/music/*.wav, and finally generated loops if both fail. Menu, combat, and shop scenes request their own looping music contexts, while SFX hooks cover menu start, combat drag swaps, combat match/combo/result/victory/defeat events, and shop purchase/reroll/sell/treasure chest success or failure feedback. AudioManagerResolver owns the shared dynamic /root/AudioManager lookup and lazy already-open-editor fallback used by menu, combat, and shop controllers. Verbose music diagnostics are opt-in through debug/audio_diagnostics_enabled. (source: scripts/core/audio_manager.gd, scripts/core/audio_stream_loader.gd, scripts/core/audio_manager_resolver.gd, resources/audio/music/, resources/audio/raw_music/, project.godot, scripts/scenes/main_menu.gd, scripts/scenes/combat.gd, scripts/scenes/shop.gd)
  • The main menu creates a local MainMenuMusicPlayer for desktop/editor WAV playback, using the same AudioStreamLoader WAV/imported-stream helpers as AudioManager. It stops retry polling after playback succeeds, stops any shared AudioManager combat/shop track before local menu playback, and reconnects through a finished-signal restart path if the stream ends unexpectedly. When Start Run is pressed on desktop/editor, the menu stops its local player and starts shared AudioManager combat music before scene transition so the combat track can continue through loading. On Android/template builds, menu music is routed through AudioManager instead, so menu/combat/shop music all use the shared WAV-first Android export path. (source: scripts/scenes/main_menu.gd, scripts/core/audio_manager.gd, scripts/core/audio_stream_loader.gd, resources/audio/music/main-menu.wav)
  • MIDI source music lives in raw/ with the GeneralUser GS SoundFont and is rendered to Godot-ready signed 16-bit WAV files by tools/audio/export_midi_to_wav.py through the local FluidSynth binary path. The script verifies WAV headers and peak-normalizes the rendered PCM data. (source: raw/, tools/audio/export_midi_to_wav.py, resources/audio/music/)

The stock/relic cards prioritize readable item names, rarity/type labels, framed art, wrapped descriptions, price badges, and explicit sold/treasure chest/gold-blocked state badges in non-overlapping bands; native stock/relic tooltips are disabled so they do not overlap the custom cards. The shop applies the shared PlayerLoadoutHud.shop_player_hud_layout_preset() with a bottom-sticky PlayerHudSection (492px tall section anchored at y=1428 in the 1080x1920 design space) connected close below the action row. The embedded Sell action appears only for the selected hovered equipment/consumable slot; shop render refreshes preserve that popover, and the shared HUD outside-click handler can activate Sell from mouse/touch hits inside the visible Sell button before applying outside-dismissal.
