# Known Issues

**Summary**: Current risks, validation gaps, and implementation mismatches that should stay visible while the prototype evolves.

**Sources**: `docs/test_plan.md`, `docs/system_architecture.md`, `docs/architecture_review_tasks.md`, `docs/tmp_transition_delay_handoff.md`, `scripts/shop/shop_service.gd`, `scripts/content/content_registry.gd`, `scripts/core/run_state.gd`

**Last updated**: 2026-05-06

---

## Overview

The prototype is functional, but several QA items remain unchecked and a few implementation gaps still need follow-up. (source: `docs/test_plan.md`)

## Details

- General combat touch selection offset was manually retested and accepted during AR-13 for real Android touch drag, rapid-tap feel, cascade feel after drag release, and board coordinate accuracy. Broader Android/on-device QA remains useful for layout, audio, and full-run feel. Shop HUD touch outside-dismissal was manually retested and confirmed during AR-04 after routing touch events through the shared HUD focus handler and refreshing selection state. (source: `docs/test_plan.md`, `docs/architecture_review_tasks.md`, `scripts/combat/board_drag_input_handler.gd`, `scripts/flow/shop_player.gd`, `scripts/ui/player_loadout_hud.gd`)
- Board lock behavior during resolution and transitions still needs full end-to-end validation. (source: `docs/test_plan.md`)
- Desktop and broad mobile overlap checks remain open in the QA checklist. The M12 reference-faithful mobile combat layout probe reports no actionable overlap and `7/7` readable-zone checks for `1080x1920`, `1080x2400`, and `900x1600`, and a Godot MCP running-scene screenshot was inspected at the current portrait runtime viewport; however, exact target-viewport screenshot review, human visual approval, and a real touch drag check are still required before the readability gate and export packaging can proceed. (source: `docs/test_plan.md`, `scripts/debug/mobile_combat_layout_probe.gd`)
- Shop mobile readability has a source-validated portrait geometry pass, but not an active in-run screenshot acceptance pass yet. The current `shop_player.gd` layout probe reports the stock row exactly fits its content width, the action row does not overlap the shared HUD, and the action-to-HUD gap is 30 design pixels; target-device visual review should still confirm card text, relic row, action buttons, and shared HUD readability in a real shop step before treating shop polish as final. (source: `docs/test_plan.md`, `scripts/flow/shop_player.gd`)
- Resolved scene-review HUD ownership risk: combat and shop now consume the same reusable `scenes/ui/player_hud.tscn` PlayerHUD component and bind the same full HUD node-key contract through `PlayerLoadoutHud`. `PlayerLoadoutHud.shop_player_hud_layout_preset()` only positions the whole shop HUD section below the action row, while internal HUD geometry/chrome remains shared and HUD-owned. `scripts/debug/player_hud_contract_probe.gd` guards against future combat/shop PlayerHUD drift. (source: `scenes/ui/player_hud.tscn`, `scripts/flow/shop_player.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/ui/player_loadout_hud.gd`, `scripts/debug/player_hud_contract_probe.gd`)
- The M12 mobile combat UI now has derived art-backed chrome and structural restage passes that move toward the reference RPG hierarchy: taller ornate header, full-width enemy backdrop with a single static foreground enemy image, smaller/readable right-side intent module, dark/gold READY divider at rest, restored larger board, broader `MASTERY` strip, player HP/vitals panel with embedded labeled relic placeholders, and a footer split into equipment and consumables only. The latest constrained pass deliberately left board, gems, READY divider, header, and screen order unchanged while improving lower-HUD spacing. A temporary spritesheet-backed enemy experiment was reverted; the accepted runtime path is the static `EnemyPortrait` layer over `EnemyStageBackdrop`. The layout probe is green for `1080x1920`, `1080x2400`, and `900x1600`, and the latest Godot MCP running-scene screenshot was rated 9/10 by the orchestrator, but exact target-device touch/layout approval remains pending. The `ruin_lancer`, `vault_executioner`, and `goldbound_keeper` stage composites are limited by low-detail source enemy art and may need generated replacements if the contact sheet is not visually accepted. (source: `docs/test_plan.md`, `scenes/combat/combat_player.tscn`, `scripts/combat/combat_layout_manager.gd`, `scripts/combat/combat_chrome_styler.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/ui/player_loadout_hud.gd`, `resources/art/first_pass/derived/combat_ui/`, `resources/art/first_pass/derived/combat_layers/`, `resources/art/first_pass/enemy_sprites/`)
- Merchant Compass free-first-reroll behavior is still unchecked in the QA plan, and `scripts/shop/shop_service.gd` does not currently implement a special free reroll path. (source: `docs/test_plan.md`, `scripts/shop/shop_service.gd`)
- Boss reward and final summary route sanity was manually accepted during AR-17 for normal victory continue, boss reward claim/skip, final boss summary, defeat summary, debug fight win/lose, and main-menu return behavior. Longer full-run playthroughs remain useful for Milestone 10 balance and content QA rather than as an open AR route blocker. (source: `docs/test_plan.md`, `docs/architecture_review_tasks.md`, `scripts/combat/combat_player_controller.gd`, `scripts/core/run_state.gd`)
- `Start Run -> Combat` is no longer blocked by combat scene instantiation after the lazy `VisualRegistry` fix: AR-01 user route-level validation from the real Start Run button measured combat resource load around `200ms`, instantiate `0ms`, attach around `84ms`, and first usable frame around `294ms`. A 2026-05-04 follow-up generated clean derived orb PNGs in `resources/art/first_pass/derived/orbs/` so `VisualRegistry` no longer needs the expensive runtime orb-sheet cleanup on the normal path; a live Start Run trace measured resource load `232ms`, instantiate `1ms`, first usable frame at `314ms`, and `combat_after_texture_map` at `325ms`. Manual visual QA is still needed for board pop-in and perceived Start Run feel on target hardware. AR-01 also measured `Combat -> Shop` at resource load around `51ms`, instantiate `0ms`, attach around `114ms`, first usable frame around `218ms`, and `Shop -> Combat` at resource load around `2ms`, instantiate `0ms`, attach around `72ms`, first usable frame around `78ms`. (source: `scripts/ui/visual_registry.gd`, `resources/art/first_pass/derived/orbs/`, `scripts/combat/combat_player_controller.gd`, `scripts/flow/shop_player.gd`, `docs/tmp_transition_delay_handoff.md`, `docs/test_plan.md`)
- External content migration remains deferred. The current prototype source of truth is dictionary-backed `ContentRegistry` content; a future `.tres`, JSON, or other data-source migration should keep the registry read API as the compatibility boundary. (source: `docs/system_architecture.md`, `scripts/content/content_registry.gd`)
- Board-debug scene coverage was retired in AR-08. Current regression coverage should rely on `combat_player.tscn`, `shop_player.tscn`, `final_run_summary.tscn`, and focused Godot MCP editor-script probes for board resolver, combat envelope, RunState routing, and content contracts. (source: `AGENTS.md`, `docs/architecture_review_tasks.md`, `docs/test_plan.md`)
- Transition ownership contract is still not fully formalized across `RunState` and scene controllers, but the scene-review P1/P2 cleanup resolved the known failure-path blockers: Start Run and final-summary New Run now prepare combat before mutating run/audio state and restore snapshots on attach failure, and shop ready-time no-active-run/wrong-step redirects now route through `RunState.flow_trace_change_scene(...)` with visible failure status. Prepared scene attach now keeps the old scene disabled/hidden until the new scene survives a deferred post-ready frame, then frees the old scene only after logging `post_ready_check`; if that health check fails, the old scene and supplied rollback snapshot are restored. Prepared scene attach and the generic `flow_trace_change_scene(...)` path both support post-ready failure callbacks and rollback snapshots, so locked main-menu, Collection, shop, final-summary, and combat callers can unlock restored UI, restore mutated run/shop state when needed, or show retryable combat routing after a deferred rollback. Combat outcome routing also checks immediate scene-change failures and restores a retryable outcome overlay instead of dropping failed route results. The attach-failure rollback snapshot now covers nested player combat state, progression state, and shop state instead of only top-level route fields. The later FlowTrace/callback cleanup added a transition generation guard so stale one-frame deferred rollback callbacks skip snapshot restoration after a newer attach, `start_new_run()`, or `reset_run()` mutation. (source: `docs/architecture_review_tasks.md`, `docs/test_plan.md`, `scripts/core/run_state.gd`, `scripts/core/main_boot.gd`, `scripts/flow/collection.gd`, `scripts/flow/final_run_summary.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/flow/shop_player.gd`)
- `combat_player_controller.gd` is still large after the AR helper extractions: 2432 lines on the AR-18 branch. The god-object risk is reduced because debug console, turn-log formatting, layout, VFX, drag input, placeholder textures, visual chrome, resolve presentation, outcome overlay presentation, and HUD snapshot building are extracted, but combat flow routing, scene-node HUD application, and turn orchestration remain in the controller and should be treated as future coordinator-boundary refactor candidates. (source: `docs/architecture_review_tasks.md`, `scripts/combat/combat_player_controller.gd`)
- Shared HUD API drift risk is narrowed after the scene-review cleanup because named HUD layout presets now live in `PlayerLoadoutHud`; future screens that need alternate HUD geometry should add shared presets there instead of screen-local internal HUD constants. (source: `docs/architecture_review_tasks.md`, `scripts/ui/player_loadout_hud.gd`, `scripts/flow/shop_player.gd`)
- Temporary diagnostics are retained intentionally for Milestone 10/M12 QA: `RunState` FlowTrace logs, combat `ResolveTrace` logs, `RunState.flow_trace_debug_snapshot()`, and the feature-flagged AR-01 combat result-envelope probe. FlowTrace route state is now capped at 50 retained routes, so the diagnostic no longer grows unbounded during long sessions. These diagnostics should be retired, feature-flagged further, or moved into a narrower debug harness if they outlive balance/build-readiness QA usefulness. (source: `docs/architecture_review_tasks.md`, `docs/test_plan.md`, `scripts/core/run_state.gd`, `scripts/combat/combat_player_controller.gd`, `scripts/debug/ar01_combat_result_probe.gd`)
- AR-01 baseline capture found repeated unsourced Godot `GDScript::reload: Integer division. Decimal part will be discarded.` warnings after scene smokes. Later AR batches repeatedly reached no-session-error `get_godot_errors` gates, but if the warning reappears it should still be localized before treating error output as a clean regression signal. (source: `docs/test_plan.md`)
- Android CLI export on this Windows checkout can hang after writing a valid `Orbwalker.apk`, leaving `Godot_v4.6.2-stable_win64_console.exe` and Java/Gradle processes alive. The observed workaround is to verify the APK timestamp/size, run `adb install -r D:\godot\matchatro\Orbwalker.apk`, then stop the stuck console exporter and Java child. Root cause is not confirmed; candidates include Gradle shutdown, the Godot 4.6.2 console exporter, or the enabled MCP editor plugin Android export warning. (source: `wiki/setup.md`, `docs/test_plan.md`)
- Balance tuning is tracked through the Milestone 10 task tracker. The M10-01 inventory confirmed the active tuning owners before value changes: shop pricing and much prototype content live in dictionary-backed `ContentRegistry`, while current runtime encounter selection and enemy/boss stats are still owned by `RunState`. `ContentRegistry` still carries older enemy rows for contract/content coverage, so active enemy HP/intent tuning should use `RunState` unless ownership is deliberately migrated later. (source: `docs/milestone_10_balance_tasks.md`, `scripts/content/content_registry.gd`, `scripts/core/run_state.gd`, `todo.md`, `docs/game_design_document.md`)
- M10-02 captured 3 human-played untuned baseline logs before tuning. The evidence showed mixed blockers: first-shop affordability could be weak, low-effectiveness play could die before shop access, and stronger runs still hit level 2-3 survivability pressure. M10-03 through M10-06 added temporary balance levers, fixed fight base rewards (`10/12/14`), first-shop `Shortsword` affordability, booster presence, equipment-heavy shop stock, rare consumables, and level/type-scoped survivability identities. M10-07 closeout compared tuned logs against the untuned baseline and found no Milestone 10 blocker: `run_1777968781_770133_2026-05-05t16_13_01` died in Dungeon 3, `run_1777969048_434533_2026-05-05t16_17_28` won a full run, and `run_1777973747_694854_2026-05-05t17_35_47` reached L3 enemy 2 after first-shop affordability, booster buys, equipment buys, and two boss relic rewards. Full-slot equipment friction is visible in late Dungeon 3 shops, but it is a post-Milestone-11 UX/economy tuning question unless player testing says it blocks the run. Deeper combat/economy rebalance is deferred until after meta progression. These values are temporary M10 playtest scaffolding, not final balance. (source: `docs/milestone_10_balance_tasks.md`, `docs/test_plan.md`, `scripts/core/run_state.gd`, `scripts/combat/combat_turn_logger.gd`, `scripts/shop/shop_service.gd`, `logs/run_1777973747_694854_2026-05-05t17_35_47.json`)
- Screenshot-reported combat victory overlay bug from 2026-05-05 was fixed for the M10-04 flow: the victory overlay now formats total fight gold as base reward plus matched gold, so a first fight with `+3` matched gold and `10` base reward displays `GOLD GAINED +13`, `Defeat enemy: 10 gold`, and `Bonus gold: 3 gold`. Manual visual QA remains useful because the focused validation covered formatter/runtime payloads rather than a screenshot pass. (source: user screenshot on 2026-05-05, `scripts/combat/combat_player_controller.gd`, `scripts/combat/combat_turn_logger.gd`, `scripts/core/run_state.gd`)
