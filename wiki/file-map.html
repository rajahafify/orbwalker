Overview

The repository is a Godot project with scenes, gameplay scripts, first-pass art, and content assets already separated into focused folders. (source: project.godot, repository layout)

Details

  • scenes/ - Scene entry points and player-facing flow. Important files include scenes/main_menu.tscn, scenes/combat.tscn, reusable board surface scenes/ui/board.tscn, scenes/shop.tscn, scenes/run_summary.tscn, scenes/collection.tscn, shared HUD scene scenes/ui/player_hud.tscn, and scenes/ui/elemental_mastery_hud_variants.tscn. Earlier empty placeholder folders under scenes/ have been pruned; root player-facing scene paths remain unchanged. The combat scene now contains explicit mobile HUD zones, a timer-only strip, a board-level outcome overlay that hands defeat to the final run summary with a Run Summary button, boss reward selection in the victory overlay, and an instanced shared PlayerHudSection; shop instantiates the same PlayerHudSection scene in code and uses the same internal HUD layout while only positioning the whole HUD below the shop action row. The Collection scene is the M11 equipment progression surface for Total Score and adjacent rarity claims. The old board-debug, boss relic reward, shop placeholder, and broken root Elemental Mastery testbed scenes were removed. The UI variant scene is a standalone Elemental Mastery visual comparison gallery. docs/scene_structure_refactor_plan.html proposes a final taxonomy that separates full screens from reusable components and development previews. (source: repository layout, docs/scene_structure_refactor_plan.html, scripts/core/run_state.gd, scenes/combat.tscn, scenes/ui/player_hud.tscn, scripts/scenes/shop.gd, scenes/run_summary.tscn, scenes/collection.tscn, scenes/ui/elemental_mastery_hud_variants.tscn)
  • scripts/core/ - Global run orchestration and shared startup logic. run_state.gd is the main autoload-backed facade for run state, public transition APIs, Run Log APIs, player-profile reset/snapshot wrappers, Run Score tracking, Total Score banking, equipment unlock APIs, victory-based equipment unlock awards, prototype balance APIs, and additive state-change signals for gold, run step, run state, profile/meta, and run summary changes. scene_router.gd owns FlowTrace route storage, retained route caps, prepared-scene loading/attach helpers, transition generation, and rollback generation guards behind the RunState facade. run_logger.gd owns passive Run Log event capture, serial allocation, snapshots, opt-in JSON/HTML/text export assembly, and export bookkeeping behind the RunState facade; run_log_reporter.gd formats readable text and HTML reports from a RunState.run_log_snapshot() payload. profile_repository.gd owns profile/meta-profile file load/save, legacy meta-profile migration, save warnings, and last-I/O probe metadata behind the RunState facade. balance_manager.gd owns prototype balance defaults/current levers, normalization, ProjectSettings sync, ContentRegistry propagation, fight rewards, starting gold, and encounter scaling behind the RunState facade. flow_result_utils.gd owns shared flow/result dictionary helpers used by player-facing scene controllers. audio_manager.gd owns music/SFX playback and generated fallback tones; audio_stream_loader.gd owns shared WAV byte loading, signed PCM16 parsing, frame-count calculation, and loop configuration; and audio_manager_resolver.gd owns the shared dynamic /root/AudioManager lookup plus lazy already-open-editor fallback. Desktop/editor playback prefers exported WAV music over code-generated fallback loops. Android/template builds try non-imported raw WAV payload copies in resources/audio/raw_music/ first, then music WAV resources, then generated fallback, so exported APK playback can keep full uploaded WAV byte length and loop timing. Main-menu scene behavior lives in scripts/scenes/main_menu.gd. (source: scripts/core/run_state.gd, scripts/core/scene_router.gd, scripts/core/run_logger.gd, scripts/core/run_log_reporter.gd, scripts/core/profile_repository.gd, scripts/core/balance_manager.gd, scripts/core/flow_result_utils.gd, scripts/scenes/main_menu.gd, scripts/core/audio_manager.gd, scripts/core/audio_stream_loader.gd, scripts/core/audio_manager_resolver.gd, resources/audio/music/, resources/audio/raw_music/)
  • scripts/board/ - Board MVC/domain scripts: BoardModel owns the 5x6 cell state and generation, BoardView owns board rendering, animation, board-local coordinate helpers, board presentation model binding, drag visuals, and board input filtering, BoardController owns BoardView.gui_input and mouse_exited wiring, board-local hover-id translation, drag/touch interpretation, swap lifecycle, board input enable forwarding, board initialization, board mutation helpers, consumable orb conversion, visual/simulation resolve model handoff, visual replay mutation APIs for clear/gravity/refill commits, and final model commit, and BoardMatchResolverService owns match/cascade resolution. (source: scripts/board/board_model.gd, scripts/board/board_view.gd, scripts/board/board_controller.gd, scripts/board/board_match_resolver_service.gd)
  • scripts/combat/ - Combat domain state plus the local combat MVC collaborators and narrow combat presentation/debug/layout/VFX/chrome/placeholder/HUD helpers. CombatController owns combat scene orchestration, resolver simulation, RunState/FlowTrace routing calls, board binding, VFX timing decisions, shared HUD payload decisions, replay timing decisions, consumable-use eligibility/status/audio consequences, and scene transitions. CombatModel owns scene-local mutable bookkeeping for input phase, external lock reason, combat_speed, FlowTrace route id, pending next-scene path, outcome transition lock, hovered board orb id, staged HUD replay values and replay mutation helpers, combat mastery preview totals/token and release bookkeeping, and resolve trace active/origin/pass state; it does not own PlayerState, EnemyState, BoardModel, CombatStateMachine, RunState, resolver math, or content state. CombatView owns combat scene node binding plus HUD snapshot application, top/enemy/timer/player/debug rendering, timer text/fill/color sync, enemy intent bubble and enemy block preview rendering, shared PlayerLoadoutHud render/control wrappers, view hover signals, visual chrome, backdrop/scrim helpers, zone guides, loadout rail layout, sole CombatLayoutPresenter construction/binding/application, enemy/player VFX target geometry helpers, and VFX presenter binding-node packaging. CombatController binds the board-domain BoardController from scripts/board/ with the current BoardModel and BoardView, and it binds PlayerState to the current run-owned mastery provider during combat setup. combat_hud_presenter.gd owns side-effect-free combat HUD snapshot dictionaries, timer state derivation, top-step labels, primary intent badge data, enemy intent preview rows, and intent damage preview dictionaries; combat_debug_command_adapter.gd owns combat debug command effects behind controller-injected orchestration callbacks; combat_consumable_service.gd owns combat-scoped consumable effect application through injected board conversion callbacks; combat_placeholder_textures.gd owns only code-generated timer, intent, enemy portrait, and hero portrait placeholder texture builders; combat_chrome_styler.gd owns code-built combat chrome and style construction; combat_layout_presenter.gd owns combat scene geometry; combat_vfx_presenter.gd owns transient combat VFX drawing mechanics; combat_debug_console.gd owns debug command parsing/dispatch and log rendering; combat_turn_log_presenter.gd owns turn-log text and outcome strings; combat_resolve_presenter.gd owns board-space resolve replay presentation mechanics; and combat_outcome_overlay.gd owns standard outcome card and boss reward overlay presentation state/layout. (source: scripts/combat/player_state.gd, scripts/combat/combat_state_machine.gd, scripts/combat/combat_model.gd, scripts/combat/combat_view.gd, scripts/combat/combat_controller.gd, scripts/scenes/combat.gd, scripts/combat/combat_hud_presenter.gd, scripts/combat/combat_debug_command_adapter.gd, scripts/combat/combat_consumable_service.gd, scripts/combat/combat_placeholder_textures.gd, scripts/combat/combat_chrome_styler.gd, scripts/board/board_controller.gd, scripts/combat/combat_layout_presenter.gd, scripts/combat/combat_vfx_presenter.gd, scripts/combat/combat_debug_console.gd, scripts/combat/combat_turn_log_presenter.gd, scripts/combat/combat_resolve_presenter.gd, scripts/combat/combat_outcome_overlay.gd)
    • Combat MVC status: scripts/scenes/combat.gd is a thin host, CombatView is now the primary combat render surface and sole CombatLayoutPresenter owner, CombatModel owns the accepted scene-local state surface plus staged HUD/mastery replay bookkeeping, CombatHudPresenter owns HUD/intent presentation data, CombatDebugCommandAdapter owns debug command effects, and CombatConsumableService owns combat consumable effect application. The 2026-05-07 compatibility and SOLID cleanups removed controller reach-ins to private view helper nodes, dead render-only wrappers, the generic CombatView.node(...) accessor, redundant controller layout-presenter storage, controller layout result mirrors, direct controller enemy/player VFX target lookup and fallback geometry, HUD/intent formatting helpers, debug command body methods, staged HUD dictionary math, shared HUD node-map assembly, controller VFX layer visibility control, normal-path shared HUD rendering/control fallback calls, render-only unused controller mirror fields for timer/enemy/chrome/layout visual nodes, normal-path controller ownership of status/turn-summary/debug/outcome UI node references, final bootstrap mirrors for background/layout root/board panel/title/hint labels, and additional controller wrapper/delegate helpers (timer sync, intent hover relays, replay VFX relay methods, dead turn-summary/outcome formatter wrappers) now called directly through CombatView and CombatVfxPresenter boundaries. CombatController still owns combat flow, input phase, RunState/FlowTrace routing, debug command callbacks, boss reward claim/skip, resolver kickoff, VFX timing, replay order, and board orchestration, with only _board and _board_view retained as direct scene-node mirrors. No new folders, scene paths, or runtime files were added for these view compatibility cleanups. (source: docs/test_plan.html, scripts/scenes/combat.gd, scripts/combat/combat_model.gd, scripts/combat/combat_controller.gd, scripts/combat/combat_view.gd, scripts/combat/combat_hud_presenter.gd, scripts/combat/combat_debug_command_adapter.gd, scripts/combat/combat_consumable_service.gd)
  • scripts/content/ - Content registry and validation. content_registry.gd is the current dictionary-backed content source of truth and compatibility API for equipment, consumables, mastery cards, treasure chests, relics, enemies, bosses, shop pools, shop pricing, validation, and future data-source migration. The internal collection/key remains treasure chests for compatibility while public copy uses treasure chest terminology. (source: scripts/content/content_registry.gd)
  • scripts/run/ - Player progression state, player progression service, persistent player profile state, and persistent meta progression state. player_profile_state.gd owns the default profile identity and contains MetaProfileState; meta_profile_state.gd stores Total Score, unlocked equipment ids, and recent equipment unlock toast payloads separately from per-run state; player_progression_service.gd enforces one equipped item per equipment family. (source: scripts/run/player_progression_state.gd, scripts/run/player_progression_service.gd, scripts/run/player_profile_state.gd, scripts/run/meta_profile_state.gd)
  • scripts/shop/ - Shop domain and scene-MVC scripts. shop_state.gd owns shop data, shop_session.gd owns the RunState-facing shop session facade/logging-hook/snapshot boundary, and shop_service.gd remains the shop economy mutation authority. shop_model.gd is the shop scene state facade, shop_view.gd owns UI/layout/rendering and shared HUD wiring, and shop_controller.gd owns shop actions, transitions, rollback callbacks, and shop audio/FlowTrace integration. (source: scripts/shop/shop_state.gd, scripts/shop/shop_session.gd, scripts/shop/shop_service.gd, scripts/shop/shop_model.gd, scripts/shop/shop_view.gd, scripts/shop/shop_controller.gd)
  • scripts/scenes/ - Thin hosts for player-facing scenes: main_menu.gd, combat.gd, shop.gd, run_summary.gd, and collection.gd. These mirror the flat root scene files under scenes/. The scene hosts instantiate local model/view/controller collaborators, bind scene nodes, and forward lifecycle or signal events into controller actions. (source: scripts/scenes/main_menu.gd, scripts/scenes/combat.gd, scripts/scenes/shop.gd, scripts/scenes/run_summary.gd, scripts/scenes/collection.gd, scripts/main_menu/main_menu_model.gd, scripts/main_menu/main_menu_view.gd, scripts/main_menu/main_menu_controller.gd, scripts/combat/combat_model.gd, scripts/combat/combat_view.gd, scripts/combat/combat_controller.gd, scripts/collection/collection_model.gd, scripts/collection/collection_view.gd, scripts/collection/collection_controller.gd, scripts/run_summary/run_summary_model.gd, scripts/run_summary/run_summary_view.gd, scripts/run_summary/run_summary_controller.gd)
  • scripts/debug/ - Retained legacy probes and debug-only validation helpers, including the AR-01 combat result-envelope probe and shared PlayerHUD contract probe. The AR-01 probe is disabled by default behind debug/ar01_combat_result_probe_enabled and is intended for Godot MCP/editor-script regression capture before architecture refactors. player_hud_contract_probe.gd verifies combat/shop keep the same scenes/ui/player_hud.tscn layout and binding keys, and now also checks shop host-to-view preload wiring plus shared HUD scene instancing from shop_view.gd. The old board-debug controller was removed in AR-08. (source: scripts/debug/ar01_combat_result_probe.gd, scripts/debug/player_hud_contract_probe.gd)
  • scripts/tests/ - Preferred home for new lightweight automated logic tests using the *_test.gd naming convention. Tests should be callable from Godot MCP/editor scripts or debug tooling and return structured pass/fail reports; use this folder for new model, state-machine, board, integration, and service tests instead of adding new test runners under scripts/debug/. Current tests cover every MVC model (BoardModel, CombatModel, MainMenuModel, ShopModel, CollectionModel, and RunSummaryModel) plus CombatStateMachine invariants; scripts/tests/integrations/board_resolver_test.gd covers board resolver integration cases, and scripts/tests/shop_view_layout_test.gd guards the accepted 9/10 shop presentation probe contract. (source: scripts/ui/visual_registry.gd, scripts/ui/visual_registry_data.gd, resources/art/first_pass/derived/orbs/, scenes/ui/player_hud.tscn, scripts/ui/player_loadout_hud.gd, scripts/ui/ui_utils.gd, scripts/ui/achievement_toast.gd, scripts/ui/elemental_mastery_hud_variants.gd)
