## Overview

The repository is a Godot project with scenes, gameplay scripts, first-pass art, and content assets already separated into focused folders. (source: `project.godot`, repository layout)

## Details

- `scenes/` - Scene entry points and player-facing flow. Important files include `scenes/main_menu.tscn`, `scenes/combat.tscn`, reusable board surface `scenes/ui/board.tscn`, `scenes/shop.tscn`, `scenes/run_summary.tscn`, `scenes/collection.tscn`, shared HUD scene `scenes/ui/player_hud.tscn`, and `scenes/ui/elemental_mastery_hud_variants.tscn`. Empty `scenes/run/` and `scenes/shop/` folders remain from earlier prototype structure. The combat scene now contains explicit mobile HUD zones, a timer-only strip, a board-level outcome overlay that hands defeat to the final run summary with a `Run Summary` button, boss reward selection in the victory overlay, and an instanced shared `PlayerHudSection`; shop instantiates the same `PlayerHudSection` scene in code and uses the same internal HUD layout while only positioning the whole HUD below the shop action row. The Collection scene is the M11 equipment progression surface for Total Score and adjacent rarity claims. The old board-debug, boss relic reward, and shop placeholder scenes were removed in AR-08. The UI variant scene is a standalone Elemental Mastery visual comparison gallery. `docs/scene_structure_refactor_plan.md` proposes a final taxonomy that separates full screens from reusable components and development previews. (source: repository layout, `docs/scene_structure_refactor_plan.md`, `scripts/core/run_state.gd`, `scenes/combat.tscn`, `scenes/ui/player_hud.tscn`, `scripts/scenes/shop.gd`, `scenes/run_summary.tscn`, `scenes/collection.tscn`, `scenes/ui/elemental_mastery_hud_variants.tscn`)
- `scripts/core/` - Global run orchestration and shared startup logic. `run_state.gd` is the main autoload-backed facade for run state, public transition APIs, Run Log APIs, player-profile reset/snapshot wrappers, Run Score tracking, Total Score banking, equipment unlock APIs, victory-based equipment unlock awards, prototype balance APIs, and additive state-change signals for gold, run step, run state, profile/meta, and run summary changes. `scene_router.gd` owns FlowTrace route storage, retained route caps, prepared-scene loading/attach helpers, transition generation, and rollback generation guards behind the `RunState` facade. `run_logger.gd` owns passive Run Log event capture, serial allocation, snapshots, opt-in JSON/Markdown/text export assembly, and export bookkeeping behind the `RunState` facade; `run_log_reporter.gd` formats readable text and Markdown reports from a `RunState.run_log_snapshot()` payload. `profile_repository.gd` owns profile/meta-profile file load/save, legacy meta-profile migration, save warnings, and last-I/O probe metadata behind the `RunState` facade. `balance_manager.gd` owns prototype balance defaults/current levers, normalization, ProjectSettings sync, ContentRegistry propagation, fight rewards, starting gold, and encounter scaling behind the `RunState` facade. `flow_result_utils.gd` owns shared flow/result dictionary helpers used by player-facing scene controllers. `audio_manager.gd` owns music/SFX playback and generated fallback tones; `audio_stream_loader.gd` owns shared WAV byte loading, signed PCM16 parsing, frame-count calculation, and loop configuration; and `audio_manager_resolver.gd` owns the shared dynamic `/root/AudioManager` lookup plus lazy already-open-editor fallback. Desktop/editor playback prefers exported WAV music over code-generated fallback loops. Android/template builds try non-imported raw WAV payload copies in `resources/audio/raw_music/` first, then music WAV resources, then generated fallback, so exported APK playback can keep full uploaded WAV byte length and loop timing. Main-menu scene behavior lives in `scripts/scenes/main_menu.gd`. (source: `scripts/core/run_state.gd`, `scripts/core/scene_router.gd`, `scripts/core/run_logger.gd`, `scripts/core/run_log_reporter.gd`, `scripts/core/profile_repository.gd`, `scripts/core/balance_manager.gd`, `scripts/core/flow_result_utils.gd`, `scripts/scenes/main_menu.gd`, `scripts/core/audio_manager.gd`, `scripts/core/audio_stream_loader.gd`, `scripts/core/audio_manager_resolver.gd`, `resources/audio/music/`, `resources/audio/raw_music/`)
- `scripts/board/` - Board MVC/domain scripts: `BoardModel` owns the 5x6 cell state and generation, `BoardView` owns board rendering, animation, board-local coordinate helpers, board presentation model binding, drag visuals, and board input filtering, `BoardController` owns `BoardView.gui_input` and `mouse_exited` wiring, board-local hover-id translation, drag/touch interpretation, swap lifecycle, board input enable forwarding, board initialization, board mutation helpers, consumable orb conversion, visual/simulation resolve model handoff, visual replay mutation APIs for clear/gravity/refill commits, and final model commit, and `BoardMatchResolverService` owns match/cascade resolution. (source: `scripts/board/board_model.gd`, `scripts/board/board_view.gd`, `scripts/board/board_controller.gd`, `scripts/board/board_match_resolver_service.gd`)
- `scripts/combat/` - Combat domain state and narrow combat presentation/debug/layout/VFX/chrome/placeholder/HUD helpers. The root combat scene controller now lives at `scripts/scenes/combat.gd`; it keeps combat orchestration, input phase authority, resolver simulation, RunState routing, privileged debug command callbacks, `/skip` route/state reset, VFX timing decisions, HUD snapshot application and shared HUD payload dispatch, timer runtime text/fill/color math, placeholder fallback decisions and scene-node assignment, and scene transitions. It binds the board-domain `BoardController` from `scripts/board/` with the current `BoardModel` and `BoardView`, and it binds `PlayerState` to the current run-owned mastery provider during combat setup. `combat_hud_snapshot_builder.gd` owns side-effect-free combat HUD snapshot dictionaries for top HUD, enemy stage, timer/tempo, player strip, and debug overlay data; `combat_placeholder_textures.gd` owns only code-generated timer, intent, enemy portrait, and hero portrait placeholder texture builders; `combat_chrome_styler.gd` owns code-built combat chrome and style construction; `combat_layout_manager.gd` owns combat scene geometry; `combat_vfx_manager.gd` owns transient combat VFX drawing mechanics; `combat_debug_console.gd` owns debug command parsing/dispatch; `combat_turn_logger.gd` owns turn-log text and outcome strings; `combat_resolve_presenter.gd` owns board-space resolve replay presentation mechanics; and `combat_outcome_overlay.gd` owns standard outcome card and boss reward overlay presentation state/layout. (source: `scripts/combat/player_state.gd`, `scripts/combat/combat_state_machine.gd`, `scripts/scenes/combat.gd`, `scripts/combat/combat_hud_snapshot_builder.gd`, `scripts/combat/combat_placeholder_textures.gd`, `scripts/combat/combat_chrome_styler.gd`, `scripts/board/board_controller.gd`, `scripts/combat/combat_layout_manager.gd`, `scripts/combat/combat_vfx_manager.gd`, `scripts/combat/combat_debug_console.gd`, `scripts/combat/combat_turn_logger.gd`, `scripts/combat/combat_resolve_presenter.gd`, `scripts/combat/combat_outcome_overlay.gd`)
  - AR closeout status: the legacy combat root controller was 2432 lines on the AR-18 branch, down from the pre-AR estimate of about 3357 lines, before the file was renamed to `scripts/scenes/combat.gd`. The remaining extraction candidates are coordinator-scale boundaries, not the original low-risk leaf helpers. (source: `docs/architecture_review_tasks.md`, `scripts/scenes/combat.gd`)
- `scripts/content/` - Content registry and validation. `content_registry.gd` is the current dictionary-backed content source of truth and compatibility API for equipment, consumables, mastery cards, boosters, relics, enemies, bosses, shop pools, shop pricing, validation, and future data-source migration. (source: `scripts/content/content_registry.gd`)
- `scripts/run/` - Player progression state, player progression service, persistent player profile state, and persistent meta progression state. `player_profile_state.gd` owns the default profile identity and contains `MetaProfileState`; `meta_profile_state.gd` stores Total Score, unlocked equipment ids, and recent equipment unlock toast payloads separately from per-run state; `player_progression_service.gd` enforces one equipped item per equipment family. (source: `scripts/run/player_progression_state.gd`, `scripts/run/player_progression_service.gd`, `scripts/run/player_profile_state.gd`, `scripts/run/meta_profile_state.gd`)
- `scripts/shop/` - Shop domain and scene-MVC scripts. `shop_state.gd` and `shop_service.gd` remain run-level data/economy authorities; `shop_model.gd` is the shop scene state facade, `shop_view.gd` owns UI/layout/rendering and shared HUD wiring, and `shop_controller.gd` owns shop actions, transitions, rollback callbacks, and shop audio/FlowTrace integration. (source: `scripts/shop/shop_state.gd`, `scripts/shop/shop_service.gd`, `scripts/shop/shop_model.gd`, `scripts/shop/shop_view.gd`, `scripts/shop/shop_controller.gd`)
- `scripts/scenes/` - Root controllers for player-facing scenes: `main_menu.gd`, `combat.gd`, `shop.gd`, `run_summary.gd`, and `collection.gd`. These mirror the flat root scene files under `scenes/`. `main_menu.gd`, `collection.gd`, `run_summary.gd`, and `shop.gd` are now thin hosts that instantiate local model/view/controller collaborators, bind scene nodes, and forward lifecycle or signal events into controller actions. (source: `scripts/scenes/main_menu.gd`, `scripts/scenes/combat.gd`, `scripts/scenes/shop.gd`, `scripts/scenes/run_summary.gd`, `scripts/scenes/collection.gd`, `scripts/main_menu/main_menu_model.gd`, `scripts/main_menu/main_menu_view.gd`, `scripts/main_menu/main_menu_controller.gd`, `scripts/collection/collection_model.gd`, `scripts/collection/collection_view.gd`, `scripts/collection/collection_controller.gd`, `scripts/run_summary/run_summary_model.gd`, `scripts/run_summary/run_summary_view.gd`, `scripts/run_summary/run_summary_controller.gd`)
- `scripts/debug/` - Resolver test runner, retained AR-01 combat result-envelope probe, and shared PlayerHUD contract probe. The AR-01 probe is disabled by default behind `debug/ar01_combat_result_probe_enabled` and is intended for Godot MCP/editor-script regression capture before architecture refactors. `player_hud_contract_probe.gd` verifies combat/shop keep the same `scenes/ui/player_hud.tscn` layout and binding keys, and now also checks shop host-to-view preload wiring plus shared HUD scene instancing from `shop_view.gd`. The old board-debug controller was removed in AR-08. (source: `scripts/debug/board_resolver_test_runner.gd`, `scripts/debug/ar01_combat_result_probe.gd`, `scripts/debug/player_hud_contract_probe.gd`)
- `scripts/ui/` - Shared presentation helpers such as the visual registry, reusable connected player HUD/loadout/mastery renderer, canonical `UiUtils.panel_style(...)` and `UiUtils.clear_children(...)` helpers, reusable bottom-right `achievement_toast.gd`, and standalone Elemental Mastery HUD variant gallery script. `visual_registry.gd` loads generated clean combat orb PNGs from `resources/art/first_pass/derived/orbs/` before falling back to runtime orb-sheet cleanup. `player_loadout_hud.gd` owns the shared combat/shop `PlayerHudSection` geometry, footer compatibility API, and optional section-position override plumbing; the actual reusable HUD node tree now lives in `scenes/ui/player_hud.tscn` and is instanced by combat and shop instead of duplicated. `ui_utils.gd` owns small shared UI helpers used by player-facing flow and shared HUD code that need explicit border/radius/margin mapping or child-node cleanup. (source: `scripts/ui/visual_registry.gd`, `resources/art/first_pass/derived/orbs/`, `scenes/ui/player_hud.tscn`, `scripts/ui/player_loadout_hud.gd`, `scripts/ui/ui_utils.gd`, `scripts/ui/achievement_toast.gd`, `scripts/ui/elemental_mastery_hud_variants.gd`)
- `tools/asset_tools/` - Utility scripts for asset extraction, cleanup, and deterministic first-pass art generation, including HUD slicing, derived icon alpha cleanup, menu chrome/icon alpha cleanup, character portrait placeholder generation, mastery reference panel/card/VFX generation, and Elemental Mastery preview panel/card chrome generation. (source: `tools/asset_tools/hud_extractor.gd`, `tools/asset_tools/clean_derived_icons.py`, `tools/asset_tools/clean_menu_art.py`, `tools/asset_tools/generate_character_placeholders.py`, `tools/asset_tools/generate_mastery_reference_assets.py`, `tools/asset_tools/generate_mastery_preview_chrome.py`)
- `resources/art/first_pass/` - First-pass backgrounds, enemy portraits, hero portraits, UI sheets, VFX, derived icons, derived clean combat orb textures, Elemental Mastery preview frame/card chrome, derived mastery panel/card chrome, derived mastery beams/shell/impacts, and the `menu/` art package for the main menu background, logo, border, button plates, stat panel, and menu symbols. The preview scene and live combat variant-5 mastery panel use the existing real mastery icon assets for card badges, while the selected combat layout uses `mastery_preview_panel_frame.png` and `mastery_preview_card_*.png` as its panel/card chrome. Character portrait placeholders now include `heroes/hero_orbwalker.png` and enemy variants for `ruin_lancer`, `vault_executioner`, and `goldbound_keeper`. (source: repository layout, `resources/art/first_pass/menu/`, `resources/art/first_pass/heroes/`, `resources/art/first_pass/enemies/`, `resources/art/first_pass/derived/icons/mastery_fire.png`, `resources/art/first_pass/derived/orbs/`, `resources/art/first_pass/derived/ui_chrome/mastery_preview_panel_frame.png`, `resources/art/first_pass/derived/vfx/`)
- `resources/content/` - Content asset folders reserved for equipment, mastery, consumables, relics, boosters, enemies, bosses, and pricing. Runtime content remains dictionary-backed inside `ContentRegistry` for this prototype phase; these folders are reserved for a later scoped migration and are not the source of truth today. (source: `docs/system_architecture.md`, `scripts/content/content_registry.gd`)
- `resources/visual/` - First-pass visual asset map and theme resource. `resources/visual/first_pass_asset_map.json` includes main-menu art mappings plus complete runtime `enemy_portraits` and shared `hero_portraits` entries used by combat and shop portrait wiring. (source: repository layout, `resources/visual/first_pass_asset_map.json`)
- `resources/audio/music/` - Godot-ready WAV music rendered from MIDI sources for menu, combat, shop, credit, and melody candidates; runtime music loading uses these WAVs before generated fallback on desktop/editor and Android/template builds. (source: `resources/audio/music/`, `tools/audio/export_midi_to_wav.py`, `scripts/core/audio_manager.gd`, `scripts/core/audio_stream_loader.gd`)
- `resources/audio/raw_music/` - Non-imported byte-for-byte WAV payload copies (`*.wav.bin`) used by Android/template `AudioManager` loading to decode full uploaded WAV headers/data inside exported packages before falling back to standard music WAV resources. (source: `resources/audio/raw_music/`, `scripts/core/audio_manager.gd`, `scripts/core/audio_stream_loader.gd`)
- `logs/` - Local generated Run Log exports. This folder is ignored by git and receives per-run JSON, Markdown, and text files when `RunState` finalizes a run while main-menu `Generate Log` is enabled. (source: `scripts/core/run_state.gd`, `scripts/scenes/main_menu.gd`, `.gitignore`)
- `raw/` - Raw source assets that are not edited by agents unless explicitly requested, currently including MIDI music sources, `GeneralUser GS v1.471.sf2` for music export, and `spash.png`, which is referenced by `project.godot` as the boot splash image. (source: `raw/`, `tools/audio/export_midi_to_wav.py`, `project.godot`)
- `tools/audio/` - Audio pipeline helpers, including the MIDI-to-WAV renderer that shells out to the local FluidSynth binary, uses the raw SoundFont, verifies WAV headers, and normalizes rendered PCM levels. (source: `tools/audio/export_midi_to_wav.py`)
- `docs/` - Human-facing design, architecture, QA docs, and architecture-maintenance tracking. `docs/architecture_review_tasks.md` tracks refactor-risk tasks, regression harness work, and current/future architecture alignment before Milestone 10 balance closure. (source: `docs/game_design_document.md`, `docs/system_architecture.md`, `docs/architecture_review_tasks.md`, `docs/test_plan.md`)
- `docs/scene_structure_refactor_plan.md` - Documentation-only proposal for finalizing `scenes/` into full-screen, shared-component, and development-preview folders after the playable flow was finalized. (source: `docs/scene_structure_refactor_plan.md`)
- `todo.md` - Milestone and scope tracker for prototype work. (source: `todo.md`)
- `AGENTS.md` - Operating rules for source/wiki precedence, development workflow, and the project-local multi-agent workflow. (source: `AGENTS.md`)
- `.codex/` - Project-local Codex configuration, including the default model setting and custom `default`, `explorer`, and `worker` agent definitions. (source: `.codex/config.toml`, `.codex/agents/default.toml`, `.codex/agents/explorer.toml`, `.codex/agents/worker.toml`)
