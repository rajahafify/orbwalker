Matchatro Architecture Overview And Decisions

Contributor-facing map for the live prototype. This page explains where gameplay commands flow, which modules own which responsibilities, and why the current contract and asset conventions exist.

Architecture Overview

The live code follows a local MVC pattern at each major screen boundary. A scene host owns the Godot scene lifecycle, resolves scene nodes, and binds a model, view, and controller. Controllers handle player commands and orchestration. Models and services own gameplay truth. Views render already-decided state onto scene nodes.

Module Owns Should Not Own Common Files
scripts/core/ Run-level singleton state, profile integration, scene routing, shared result boundaries. Screen-local layout or per-scene rendering details. run_state.gd
scripts/run/ Dungeon sequence, progression rewards, victory/defeat transitions, run service helpers. Board matching or combat damage math. player_progression_service.gd
scripts/board/ Board model state, movement legality, match resolution, gravity, refill, board view coordination. Combat reward routing or shop inventory decisions. board_model.gd, board_resolver.gd, board_view.gd
scripts/combat/ Combat state machine, input orchestration, enemy intent, resolve timing, overlays, combat-specific services. Persistent run routing decisions or generic shared HUD internals. combat_controller.gd, combat_model.gd, combat_view.gd
scripts/shop/ Shop offers, pricing, buy/reroll/sell commands, shop rendering and shop flow handoff. Combat board resolution or content storage backend migration logic. shop_controller.gd, shop_model.gd, shop_view.gd
scripts/ui/ Shared presentation widgets, HUD surfaces, visual lookup helpers, reusable rendering helpers. Gameplay rule changes or command authorization. player_loadout_hud.gd, visual_registry.gd

Decision Records

ADR-001: Use Declarative Root Node Bindings

Status: Accepted

Context. Combat scene binding used to be mirrored manually across node declarations and root-node dictionary construction. That made layout edits tedious and easy to desynchronize.

Decision. Scene hosts declare a single binding map, such as ROOT_NODE_BINDINGS in scripts/scenes/combat.gd, and generate the dictionary passed into the existing CombatController.bind(host, root_nodes, model, view) interface.

Consequences. Node ownership stays visible in one place, while the controller interface remains stable. Every binding must also declare its expected class in ROOT_NODE_TYPES so type mistakes fail at the scene boundary.

Validation. res://scripts/tests/combat_scene_contract_test.gd instantiates res://scenes/combat.tscn, resolves the binding map, checks expected node types, and verifies BoardView resolution.

ADR-002: Scene Files Own Persistent Signal Connections

Status: Accepted

Context. Persistent header controls were previously wired in both combat.tscn and combat scene code, creating mixed ownership and duplicate connection risk.

Decision. Scene-owned controls keep persistent connections in the .tscn. Runtime-created nodes connect signals in code at the point where the node is created or bound.

Consequences. Signal ownership can be found from the scene file for persistent UI and from the creator code for dynamic UI. If ownership moves, remove the old connection and add or update a contract test in the same change.

Validation. combat_scene_contract_test.gd asserts the combat help, debug toggle, and settings buttons are connected from scenes/combat.tscn.

ADR-003: Prefer Runtime Contracts Over Source-Grep Contracts

Status: Accepted

Context. A source-grep contract can fail when a private helper is renamed even though behavior remains correct, or pass while runtime behavior is broken.

Decision. Tests and probes should assert runtime values, instantiated scenes, loaded scripts, exposed accessor dictionaries, and behavior. Avoid FileAccess.get_file_as_string() checks that search source text for private implementation details.

Consequences. Refactors can rename private code without breaking unrelated checks, while real contract regressions still fail. When a test needs ownership information, expose a small script-level accessor instead of reading another file as text.

Validation. The aggregate headless runner includes runtime scene and HUD probes, and the roadmap tracks that no contract under scripts/tests/ or scripts/debug/ relies on source-grep checks.

ADR-004: Resolve Assets Through Maps, Manifests, And Declared Fallbacks

Status: Accepted

Context. The project has generated art, first-pass derived art, runtime manifests, and explicit fallback textures. Broken paths are easy to miss visually and expensive to debug late.

Decision. Runtime texture resolution must flow through owning helpers: MainMenuModel.menu_texture_paths() for the main menu and VisualRegistry for shared visual surfaces. Stable fallback paths are declared by the owner and exposed through asset_contract_paths().

Consequences. Views receive resolved paths or textures instead of knowing the fallback order. Asset migrations can repoint maps and manifests while tests still prove the fallback inventory imports.

Validation. res://scripts/tests/asset_pipeline_test.gd validates resources/visual/first_pass_asset_map.json, resources/art/assetgen/runtime/manifest.json, assets.json, and the fallback texture contracts exposed by their owning scripts.

ADR-005: Keep Views Focused On Rendering And Layout

Status: Accepted

Context. CombatView is large because it owns many scene surfaces, but moving gameplay decisions into it would make combat behavior harder to test and reason about.

Decision. CombatView should stay focused on node rendering, layout, and visual state. CombatController keeps state-machine orchestration and delegates clear side surfaces, such as debug console and settings command handling, to helpers when ownership is obvious.

Consequences. View extraction work should target presentation subcomponents, not gameplay rules. Controller extraction work should move cohesive orchestration helpers without changing public gameplay behavior.

Validation. Flow tests under scripts/tests/integrations/run_flow_test.gd cover combat/shop routing, while the script-size budget warns when large surfaces need a documented extraction path.
