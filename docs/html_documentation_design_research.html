Research brief

Designing HTML reports and wiki pages people will actually use

Research synthesis for Matchatro's browser-first documentation system: what good reports, wiki pages, runbooks, QA reviews, and implementation plans should look like now that HTML is the default human-facing format.

User-need first Scannable by default Structured by page type Accessible and printable Reusable template system

Executive summary

Good HTML documentation is not just prettier Markdown. It should use browser-native structure to reduce reader effort: clear page purpose, strong headings, tables for comparison, callouts for risk, diagrams for relationships, and a stable navigation frame. The research points to one central rule: design the page around the user's task, then choose the component patterns that make the task easier.

The best foundation for Matchatro is a small documentation design system: a reusable HTML shell, a page-type taxonomy, a component set, and quality checks. That gives future reports a consistent look without making every agent invent layout from scratch.

Recommendation: implement a reusable Matchatro HTML documentation template and page-type guidance, then migrate future reports/wiki pages onto that style as they are touched.

Design principles

Start with the user needDefine who is reading, what decision they need to make, and what task the page helps complete.
Make scanning the first-class pathUse summaries, headings, tables, badges, and short chunks so readers can locate the right section before reading deeply.
Separate page typesDo not mix tutorial, how-to, reference, and explanation goals without labeling the mode clearly.
Put evidence next to claimsUse source paths, validation outputs, screenshots, and status metadata near the conclusion they support.
Design for maintenanceInclude owner, freshness, source of truth, changelog, and open questions so stale pages are visible.
Preserve accessibilityUse semantic HTML, readable contrast, keyboard-friendly controls, responsive layout, and print-safe styling.

Information architecture

The wiki should behave like a map, not a pile of pages. Matchatro should keep a small number of durable hubs and make every page declare where it belongs.

Hub What belongs there Recommended page fields
Gameplay systems Combat, shop, run flow, progression, content registry. Current behavior, source files, invariants, validation scenes.
Assets and visual production Assetgen workflow, candidates, QA, approvals, integration state. Asset ID, lifecycle status, source image, approved output, QA evidence.
Architecture Runtime ownership, scene structure, services, autoloads, MVC boundaries. Responsibility map, data flow, known debt, future migration notes.
Release and platform Android builds, itch readiness, packaging, device validation. Build command, artifact path, install status, device evidence, rollback.
Agent workflow Multi-agent flow, Godot MCP validation, HTML doc rules, handoff templates. Role ownership, required tools, allowed edits, validation requirements.

Visual system recommendations

  • Use a stable shell: hero/summary, main content, sticky table of contents, print styling, and responsive single-column mobile layout.
  • Use color for meaning: status, risk, priority, confidence, and category. Do not decorate with arbitrary palettes.
  • Keep line length comfortable: prose should sit in a readable column; tables and evidence can overflow horizontally inside framed regions.
  • Prefer tables for comparison: options, risks, file responsibilities, test matrices, metadata fields, and source synthesis.
  • Use callouts sparingly: warnings, decisions, blockers, assumptions, and recommendations should stand out.
  • Use diagrams when structure matters: inline SVG or Mermaid-style rendered diagrams for flows, dependencies, and state transitions.
  • Make freshness visible: include created date, last reviewed date, validation date, owner or agent role, and source-of-truth fields.

Quality checklist

Gate Pass condition
Purpose The page says who it is for, why it exists, and what decision or task it supports.
Scan path A reader can understand status and next action from headings, summary, badges, and tables in under one minute.
Evidence Claims cite source paths, commands, screenshots, validation notes, or external references.
Accessibility Semantic headings, sufficient contrast, non-color labels, keyboard-safe controls, and print-safe styles are present.
Maintainability The page includes last updated context, open questions, and LLM reuse notes for future agents.
Scope control The page separates facts, assumptions, recommendations, and decisions.

Implementation options

Option What changes Pros Tradeoffs
A Guidance only Add a concise HTML documentation design standard to AGENTS.html and wiki guidance. Low risk, fast, keeps agents aligned. Future pages may still drift visually.
B Template kit Create reusable single-file HTML templates for report, wiki, QA, decision, and runbook pages. Best balance of consistency and flexibility. Requires maintaining templates as project needs evolve.
C Generator script Add a tool that creates a new HTML doc from page type metadata. Highest consistency; reduces repetitive manual scaffolding. More tooling surface; may be overkill before the template style stabilizes.
D Full wiki restyle Rewrite all existing converted HTML docs onto the new style immediately. Maximum visual consistency now. Large diff, high review burden, low immediate product value.
Recommended path: choose Option B first. Add templates and guidance now, then migrate high-traffic pages opportunistically. Avoid a full wiki restyle until the template has survived real use.

Proposed Matchatro standard

  1. Every human-facing page starts with title, purpose, audience, status, and one-paragraph executive summary.
  2. Every long page includes a sticky or clearly visible table of contents.
  3. Every factual claim that matters cites source paths, validation notes, or external references.
  4. Every page has a page type: report, wiki, QA, decision, architecture, implementation plan, runbook, or asset review.
  5. Every durable page includes open questions and LLM reuse notes.
  6. Every interactive page includes a no-script fallback or still presents its core information without interaction.
  7. Every template is self-contained: embedded CSS, no remote scripts, no external dependencies unless explicitly requested.
