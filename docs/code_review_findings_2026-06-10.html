P0

Foundational integrity

Guards must actually guard. A quality gate that silently skips the worst offenders is worse than no gate, because it certifies health it never measured.

R-P0-1 Script-size budget guard has blind spots

Finding: tools/quality/script_size_budget.gd only scans scripts/combat, scripts/shop, scripts/collection, and scripts/run_summary (SCAN_ROOTS, lines 3–8). The four largest non-test, non-combat scripts in the repo therefore never appear in the budget report: scripts/ui/player_loadout_hud.gd (1,924 lines), scripts/ui/visual_registry.gd (1,723), scripts/core/run_state.gd (1,538), and scripts/main_menu/main_menu_view.gd (930). Two of these are larger than every documented combat exception except the controller itself. The CI log prints scanned=118 against 165 non-test scripts.

What should be done
  • Add res://scripts/ui, res://scripts/core, res://scripts/main_menu, and res://scripts/board to SCAN_ROOTS.
  • Document the four newly-visible over-budget files as exceptions with owners and reasons, exactly as the five combat exceptions are documented today.
  • Extend script_size_budget_test.gd with a case asserting the scan covers every non-test directory under scripts/, so a future folder can't silently escape.
Acceptance: budget report scans all non-test script roots; undocumented=0 still holds; new exceptions carry owner + reason; coverage test fails if a scripts/ subfolder is missing from the scan.

P1

Structural debt — the god objects

Continue the decomposition that is already working in combat, and extend it to the two large surfaces it hasn't reached: the UI layer and the RunState autoload.

R-P1-1 combat_controller.gd remains the largest single risk

Finding: 1,962 lines against a 400-line budget, preloading ~37 collaborator scripts. The exception is documented (owner P1-1) and the recent commit history shows active extraction (outcome routing, turn resolution, HUD stage coordinators all landed recently), so this is known, fenced debt — but it is still the file where a regression is most likely to hide.

What should be done
  • Keep extracting along the existing seams; the remaining in-controller candidates are replay sequencing (TURN_REPLAY_* constants and their consumers), combo popup presentation, and input-phase gating (InputPhase enum).
  • Each extraction follows the established pattern: small collaborator + dedicated run_all() test suite, landed in the same commit.
  • Ratchet the documented exception's line count down with each extraction so the budget exception cannot quietly grow back.
Acceptance: controller below ~350 lines per the roadmap's P1-1 target; documented exception removed; combat flow tests stay green.
scripts/combat/combat_controller.gd

R-P1-2 RunState autoload mixes too many roles

Finding: scripts/core/run_state.gd (1,538 lines) owns run progression, gold/score, the level sequence, profile persistence, user-settings file I/O (user://matchatro_settings.cfg), scene route constants, flow tracing, and logging. Its collaborator members are untyped (var player_state, var content_registry, …), so none of its call sites get static checking.

What should be done
  • Extract a UserSettingsStore for the cfg read/write (the USER_SETTINGS_* constants and their accessors) — it is the most self-contained slice and shop/combat/menu all consume it.
  • Move scene path constants next to scene_router.gd, which already exists as the routing seam.
  • Type the collaborator members (var player_state: PlayerState, …) — these classes already have class_name declarations or preloaded script constants.
Acceptance: settings I/O lives in a tested store; RunState members typed; RunState enters the size-budget scan (R-P0-1) with a shrinking documented exception.
scripts/core/run_state.gdscripts/core/scene_router.gd

R-P1-3 Unscanned UI god objects

Finding: scripts/ui/player_loadout_hud.gd (1,924 lines) and scripts/ui/visual_registry.gd (1,723 lines) are the second- and third-largest production scripts in the repo and have never been subject to the budget or the decomposition program. visual_registry.gd in particular looks like a static asset-path catalog that could largely become data (the same observation the budget doc already makes for combat_max_vfx_overlay.gd's catalogs).

What should be done
  • Move visual_registry.gd's static path/texture catalogs into .tres resources or a generated data file, leaving a thin typed lookup API.
  • Split player_loadout_hud.gd along its visible seams (equipment rail, consumable rail, mastery icons) using the presenter pattern combat already established.
  • Bring both under the size budget with documented exceptions first (R-P0-1), then ratchet down.
Acceptance: both files visible in the budget report; catalogs out of code; each extracted presenter has its own test suite.

P2

Contracts & type safety

The riskiest boundaries are exactly the ones typed Variant today.

R-P2-1 Variant-typed collaborators in the combat controller

Finding: combat_controller.gd holds its most complex collaborators untyped: var _combat: Variant, var _debug_runtime: Variant = null, var _settings_command_handler: Variant = null, var _combat_timer_service: Variant = null, var _resolver: Variant. Any renamed or removed method on these classes becomes a runtime error instead of a parse error — in the one file least able to afford it. The live editor session also reported two "integer used where enum expected" analyzer warnings, the same class of silent coercion.

What should be done
  • Give the collaborators class_name declarations where missing and type the fields. Where a preload cycle prevents it, type against a minimal interface-style base class instead of falling back to Variant.
  • Fix the two int-vs-enum sites with explicit as casts or proper enum values.
  • Apply the same pass to RunState's members (tracked under R-P1-2).
Acceptance: zero Variant-typed collaborator fields in combat_controller; analyzer reports no int-vs-enum warnings on reload.

P3

Test depth & signal quality

The suite is broad and green; the remaining work is making its output trustworthy enough that a new warning is a signal, not noise.

R-P3-1 Board resolver tests spam expected warnings

Finding: Six cases in scripts/tests/integrations/board_resolver_test.gd intentionally call resolve_all with max_steps=1, and board_match_resolver_service.gd:56 emits push_warning("BoardMatchResolverService hit max_steps=1 before stabilizing") each time. Six identical warning+backtrace blocks per run bury any genuine warning in the CI log.

What should be done
  • Add a suppress_warnings (or expected_truncation) parameter to resolve_all that the tests pass, or have the service return the truncation fact in its result dictionary and let real callers decide whether to warn.
  • Goal state: a fully green run prints zero warnings, so any warning in CI is actionable.
Acceptance: green headless run emits no BoardMatchResolverService warnings; the warning still fires for real gameplay truncation.
scripts/board/board_match_resolver_service.gdscripts/tests/integrations/board_resolver_test.gd

R-P3-2 Resource leaks at test-runner exit

Finding: The headless run exits with 1 RID allocation of type RendererSceneCull::Instance leaked, one Camera RID, one CanvasItem RID, and an ObjectDB instance leak. Some test or probe instantiates scene/canvas objects and never frees them. Harmless today, but it normalizes leak output — a future real leak in VFX overlays (which allocate heavily) would look identical.

What should be done
  • Run the suite once with --verbose to identify the leaking suite (the camera RID suggests a 3D/viewport-using probe), then free the nodes in that suite's teardown.
  • Once clean, consider asserting on leak output in CI (grep the runner log for leaked) so it stays clean.
Acceptance: headless exit prints no leaked RID/ObjectDB lines; CI fails if leak lines reappear.

P4

Tooling & developer experience

A quiet editor log is the cheapest quality tool there is.

R-P4-1 Stale temp probe breaks editor startup log

Finding: tmp/desktop_layout_probe.gd (local-only; tmp/ is gitignored and the hygiene test forbids tracking it) calls build_desktop_layout_probe() on ShopView, RunSummaryView, and MainMenuView — static methods that no longer exist. The editor logs 7 parse errors on every startup. Confirmed live via MCP on 2026-06-10.

What should be done
  • Delete tmp/desktop_layout_probe.gd (it was a one-off desktop-viewport experiment; the per-view probes it called were removed).
  • If desktop layout probing is still wanted, rewrite it against the current API — only CombatLayoutPresenter.build_layout_probe() survives.
  • Optionally add tmp/ to a .gdignore so the editor never parses scratch scripts.
Acceptance: editor startup log shows zero parse errors.
tmp/desktop_layout_probe.gd

R-P4-2 Analyzer warning sweep in combat VFX/view scripts

Finding: The live editor session reports ~25 GDScript analyzer warnings concentrated around the VFX/view layer: layer_size parameters shadowing a function (8 sites), shadowed locals (quality, preview, created_unix, updated_unix, item_id), unused class variables (_background, _next_button, _debug_toggle_button, _debug_overlay, _intent_badge, _player_hp_bar, _player_portrait), and unused parameters in spawn_bar_infusion(). The unused class variables look like dead bindings left behind by the recent presenter extractions.

What should be done
  • Delete the dead class-variable bindings — they imply node references that no longer have owners and will confuse the next extraction.
  • Rename shadowing locals/parameters (e.g. layer_size → size param or rename the function) and underscore-prefix intentionally unused parameters.
  • After the sweep, treat new analyzer warnings as review-blocking in the same spirit as the size budget.
Acceptance: script reload in the editor produces zero analyzer warnings across scripts/combat and scripts/ui.

Resilience & forward compatibility

Long-horizon items: not needed for the Android release, cheap to note now.

R-P5-1 No keyboard/controller focus path

Finding: Simulated ui_down/ui_accept input against the running main menu did nothing — no button takes initial focus, so the game is unreachable without touch/mouse. Reasonable scope for a touch-first Android title, but it limits desktop/itch.io play and accessibility, and main_menu_accessibility_test.gd currently doesn't cover focus navigation.

What should be done
  • Decide explicitly: if desktop/itch is on the roadmap (the itch readiness docs suggest it is), set focus_neighbor/grab_focus() on the menu buttons and extend the accessibility test to walk the focus chain.
  • If touch-only is the deliberate scope, record that as a decision in docs/architecture_decisions.html so it stops resurfacing in reviews.
Acceptance: either a focus chain exists and is tested, or the scope cut is documented.
scripts/main_menu/main_menu_view.gdscripts/tests/main_menu_accessibility_test.gd

R-P5-2 Deprecated graph_offset in a visual shader resource

Finding: Loading VFX assets during combat_model_test.gd's asset-availability case (line 353) triggers WARNING: graph_offset property is deprecated. Setting it has no effect from a saved VisualShader resource. Cosmetic on 4.6, but deprecated properties are removal candidates in future Godot majors and will spam every asset-touching test until fixed.

What should be done
  • Identify the shader (re-run the suite; the warning's backtrace points at the asset list in the test), open it once in the editor, and re-save — Godot drops the deprecated property on save.
  • Sweep other .tres shader resources for the same property while at it (grep -l graph_offset resources/ assets/).
Acceptance: headless run emits no deprecation warnings.
