SOLID → How Each Principle Applies Here

SSingle Responsibility The core violation in all five files. Every extraction below is an SRP cut: one class, one reason to change. Target budget: ≤ 400 lines / ≤ 30 functions per class.
OOpen–Closed Adding a new shop offer type, VFX effect, or settings row must not require editing a 1,900-line hub. We replace if/elif dispatch with registries and per-type handler classes. Current status: VFX routes, settings rows, and shop offer types now have registry-backed extension paths; the remaining shop work is policy dependency cleanup, not offer-type dispatch.
LLiskov Substitution Extracted services share base interfaces (e.g. CombatPresenter). Any presenter must be swappable in tests with a stub without the controller noticing.
IInterface Segregation Consumers currently depend on all of RunState to use one slice of it. We expose narrow facades (WalletService, ShopService) so the shop UI never sees profile code.
DDependency Inversion High-level flow (controller) depends on injected abstractions, not on concrete autoload singletons reached via get_node("/root/…"). The existing _bind_* pattern in the controller is the right seam — extend it.
GDScript note: Godot has no formal interfaces. We approximate them with base classes (class_name + overridable methods) or duck-typed contracts documented at the top of each service. Liskov/ISP still apply — they govern the shape of the contract, not the language feature.

The Extraction Playbook

Every extraction in this document follows the same five steps. This is what makes the refactor safe and reviewable.

  1. Characterize first. Before touching code, make sure the behavior being moved is covered by a test (extend scripts/tests/; combat_model_test.gd shows the house style). If no test exists, write a characterization test against the current behavior.
  2. Extract the class, keep the facade. Move the methods into a new class; the god object keeps thin one-line delegating methods so zero call sites change in the same PR. (Strangler pattern — the god object becomes a facade that shrinks over time.)
  3. Inject, don't reach. The new class receives its dependencies via constructor/bind() parameters — never via get_node("/root/…") or autoload references inside the class (DIP). The composition root (autoload _ready or controller _bind_*) does the wiring.
  4. Migrate call sites in a follow-up PR. Point callers at the new service, then delete the delegating methods from the facade.
  5. Ratchet. After each extraction, the god object's line count must go down. Never add new features to a file on this list — new code goes into the extracted services.

Strategy 1 — run_state.gd (highest value, do first)

CRITICAL Decompose the autoload into six injected services SRP ISP DIP

scripts/core/run_state.gd — 1,538 lines, 152 functions, 39 vars

Why first: it's an autoload everything depends on, so every line removed here lowers coupling project-wide. Its seams are also the cleanest — the shop/economy block barely touches the profile block.

Before — one class, six jobs
  • 10× ensure_* service-locator factories
  • Wallet: add_gold / spend_gold / can_afford
  • Shop: open/reroll/buy/sell/close + treasure chests + relic offers
  • Profile & meta-profile: reset/create/snapshot
  • Unlocks: unlock_equipment / claim / consume_recent
  • Balance levers: snapshot/set/reset
  • Run lifecycle: start_new_run / start_tutorial_run / reset_run
  • Transition snapshots: 3 paired snapshot/restore sets
After — RunState as thin composition root
  • core/wallet_service.gd — gold only (~80 lines)
  • shop/shop_session.gd — shop lifecycle, with its dependencies made explicit (see dependency map below; merge with existing shop_service)
  • core/profile_service.gd — profile + meta-profile + unlocks (~300 lines)
  • core/balance_levers.gd — prototype balance config (~100 lines)
  • core/run_lifecycle.gd — new run / tutorial run / reset (~150 lines)
  • core/transition_snapshot.gd — all snapshot/restore pairs (~200 lines)
  • RunState keeps: wiring + ~20 delegating methods during migration (< 300 lines at end state)

ShopSession dependency map (do this before extracting). The existing shop_service.gd takes the whole RunState node and reaches into seven things through it: ShopState (via ensure_shop_state), ContentRegistry, ShopOfferPolicy (already owned), gold mutation including refunds (spend_gold / add_gold("shop_refund")), run_gold for result reporting, PlayerProgressionState + PlayerProgressionService for buy/sell application, and relic-offer tracking (set_relic_offer_id_for_level, dungeon_level). "Depends only on wallet + content registry" is therefore not the target — if followed literally it forces either a hidden RunState back-reference or duplicated shop state. The actual goal is to make these seven dependencies explicit constructor parameters:

# shop_session.gd — every current reach-in becomes a named, injected dependency
func _init(wallet: WalletService, content: ContentRegistry, shop_state: ShopState,
        offer_policy: ShopOfferPolicy,
        progression_state: PlayerProgressionState,  # buy/sell mutate the loadout…
        progression_service: PlayerProgressionService,  # …through the service, on this state
        relic_tracker: RelicOfferTracker) -> void: ...

Refunds stay inside WalletService (spend/refund with reason strings), run_gold reporting reads from the wallet, and set_relic_offer_id_for_level moves into a small RelicOfferTracker (or lives on ShopState). If a dependency can't be named, that's a signal the seam isn't ready — map it first, extract second.

ShopOfferPolicy needs the same treatment — don't let the god object smuggle back in through the policy layer. Every public method of shop_offer_policy.gd (item_offers, relic_offer, first_shop_guarantee_entry…) also takes run_state: Node and reaches through it for: ensure_shop_state().next_offer_id(), relic-offer tracking (relic_offer_id_for_level / set_relic_offer_id_for_level), equipped-equipment and owned-relic lookups from progression state, unlock checks, shop ordinal, and prototype reward levers — and it even forwards run_state into content.shop_item_pool(level, run_state). If ShopSession gets clean constructor injection but keeps passing a RunState node into the policy, the refactor changes nothing. The three mutations the policy currently performs inline move out as follows: offer ids — the policy returns offers without ids and ShopSession mints them via shop_state.next_offer_id() when applying the result (alternatively, preallocate a batch of ids into the context; returning id-less offers is simpler and keeps the context read-only). Stale relic-cache clear — the stale-id clear at shop_offer_policy.gd:104 becomes the clear_cached_relic_id flag on the result, applied by the session to the RelicOfferTracker. New relic-cache write — when no cached id exists, relic_offer() currently picks a relic and writes it back via set_relic_offer_id_for_level (shop_offer_policy.gd:105+) so the same offer persists across shop refreshes; that selection comes back as cache_relic_offer_id and the session pins it in the tracker — the persistence guarantee is preserved, just applied in one place. With all three out, the policy is a pure function of context + content + RNG — trivially unit-testable. ContentRegistry.shop_item_pool must take the context (or the specific fields it uses) instead of the run_state node, for the same reason.

ISP payoff: shop UI takes a ShopSession, combat reward code takes a WalletService, main menu takes a ProfileService. No consumer sees the other five services.

# run_state.gd — end state: composition root only
var wallet: WalletService
var profile: ProfileService
var shop: ShopSession

func _ready() -> void:
    wallet = WalletService.new(_run_logger)
    profile = ProfileService.new(_profile_repository)
    # DIP: all mapped deps injected, not reached for (see ShopSession dependency map above)
    shop = ShopSession.new(wallet, content_registry, shop_state, offer_policy,
            progression_state, progression_service, relic_tracker, _run_logger)

# temporary facade kept during migration (step 2 of playbook):
func spend_gold(amount: int) -> bool: return wallet.spend(amount)

Extraction order (one PR each):

  1. WalletService — smallest, most-referenced; proves the pattern.
  2. ShopSession — inject the seven mapped dependencies above; fold in treasure chest and relic-offer methods.
  3. TransitionSnapshot — mechanical move of the 6 snapshot/restore methods.
  4. ProfileService + BalanceLevers + RunLifecycle.
  5. Delete delegating facades; RunState is now wiring only.

Strategy 2 — combat_controller.gd

CRITICAL Finish the coordinator extraction already in flight SRP OCP DIP

scripts/combat/combat_controller.gd — 1,962 lines, 190 functions, 47 vars

Good news: the architecture is already half-right. The _bind_* methods (_bind_audio_cue_player, _bind_debug_state_provider, _bind_combat_vfx_presenter…) are dependency injection seams, and commit 43d82ce already extracted outcome-routing coordinators. The strategy is to keep pulling on that thread until the controller is only a composition root + signal router.

Remaining responsibility clusters
  • ~14 _on_settings_* callbacks (speed, quality, reduced motion, juice flags, defaults)
  • Debug: console, overlay toggle, regenerate/print-board, run-tests, test-item injection
  • Tutorial-end continue/main-menu flow
  • Drag/swap state (_drag_* family)
  • Turn preview + combat state init
  • Viewport/input handling
Migration map — extend existing seams, don't create parallel ones
  • Settings callbacks → existing CombatSettingsCommandHandler + CombatSettingsOverlayCoordinator; the controller's _on_settings_* bodies move there, leaving connect-only stubs
  • Debug cluster → existing CombatDebugRuntime, CombatBoardDebugCommandHandler, CombatDebugCommandAdapter; remaining inline debug logic (overlay toggle, test-item buttons, run-tests) migrates into them
  • Tutorial-end flow → existing CombatTutorialEndCommandHandler + CombatTutorialEndOverlayCoordinator
  • Input/viewport handling → existing CombatInputCommandHandler
  • Drag/swap → new combat/board_drag_coordinator.gd (no existing owner for the _drag_* family)
  • Turn preview + state init → new combat/combat_session.gd (pure logic, headless-testable)

The codebase already has the right abstractions — the controller's problem is that it kept the method bodies after the handlers were introduced. Phase 2 is therefore mostly migration into existing classes, not new architecture: audit each controller method, find its rightful existing owner from the map above, and only create a new class for the two clusters with no owner.

Liskov/DIP rule for coordinators: each gets a bind(deps) → void and teardown() → void contract matching the existing handler/coordinator pattern, so the controller treats every collaborator uniformly and tests can substitute stubs.

End state: controller ≈ 400 lines — _ready wiring, signal connect/disconnect, and nothing with game logic in it. The test for "is it done": no method in the controller contains an if about game rules.

Strategy 3 — player_loadout_hud.gd

HIGH Split into slot components + a thin coordinator SRP LSP ISP

scripts/ui/player_loadout_hud.gd — 1,924 lines, 134 functions, 34 vars, 8 signals

A HUD this size is almost always several widgets trapped in one script. Decompose along the natural UI seams — Godot scenes are the cheapest module boundary in the engine, so each piece becomes its own .tscn + script:

  • ui/loadout/equipment_strip.gd + scene — renders equipment slots, emits slot_pressed(index)
  • ui/loadout/consumable_strip.gd + scene — same contract as the equipment strip (LSP: both extend a common ItemStrip base so the HUD treats them identically)
  • ui/loadout/item_tooltip_presenter.gd — tooltip build/show/hide; takes item data in, knows nothing about slots
  • ui/loadout/loadout_drag_handler.gd — drag-and-drop + sell-drop logic
  • ui/loadout/loadout_animator.gd — acquire/consume/unlock animations
  • player_loadout_hud.gd shrinks to a coordinator that wires the strips to game state and re-emits its existing 8 public signals (external API unchanged → no call-site churn)

ISP: the strips receive plain item-view-model arrays (Array[Dictionary] or a small ItemViewModel resource), not RunState. Only the coordinator knows where data comes from — which also makes the strips reusable in the shop screen.

Strategy 4 — visual_registry.gd

HIGH Data goes to resources, recipes go to stylers SRP OCP

scripts/ui/visual_registry.gd — 1,723 lines, 120 functions, 39 vars

A registry should be a lookup table, not a style factory. Two cuts:

  1. Data → resources. Palette constants, texture maps, and per-item visual definitions move into .tres custom resources (VisualTheme, OrbTextureSet). The registry keeps only get_* lookups + caching (~300 lines). OCP: adding a new item's visuals becomes adding a resource file — zero code edits.
  2. Style recipes → per-screen stylers. Functions that construct StyleBoxes/themes for a specific screen move next to that screen (the codebase already has the pattern: combat_chrome_styler.gd). Shared builders go in ui/style_builders.gd as static helpers.

Guardrail: after the split, the registry may not new() any StyleBox — construction belongs to stylers, lookup belongs to the registry.

Strategy 5 — combat_max_vfx_overlay.gd

MEDIUM Finish the migration to the existing presenter/recipe pattern SRP OCP LSP

scripts/combat/combat_max_vfx_overlay.gd — 1,088 lines, 131 functions, 40 vars

The pattern already exists — finish migrating to it. The repo has ~25 CombatMaxVfx*Presenter classes (fire/ice/earth recipe presenters, burst particles, camera kick, coin rain, flipbooks, cleanup, …) plus CombatMaxVfxAssetCatalog and CombatMaxVfxElementalRecipePolicy. The overlay's problem is not a missing abstraction; it's that 131 functions of spawn/tween/cleanup logic stayed inline in the overlay instead of moving into presenters. Do not introduce a new VfxEffect hierarchy alongside the presenter/recipe pattern — that would leave two competing abstractions.

Strategy, per the playbook:

  1. Audit each overlay function: does an existing presenter own this effect family? If yes, move the body there. If no, create a new CombatMaxVfx*Presenter following the existing naming and structure.
  2. Formalize the implicit contract the presenters already share, so dispatch is uniform and LSP holds. Make completion and cancellation concrete — not a vague signal placeholder:
# combat/combat_max_vfx_presenter_base.gd — codifies what presenters already do
class_name CombatMaxVfxPresenterBase
signal effect_finished(effect_id: String)

func bind(host: CanvasItem, catalog: CombatMaxVfxAssetCatalog, profile: CombatVfxProfile) -> void: pass
func play(effect_id: String, ctx: Dictionary) -> void: pass  # emits effect_finished when done
func cancel_all() -> void: pass  # kills tweens/particles, frees spawned nodes, no signal

Callers that need to await completion connect to effect_finished (or await presenter.effect_finished); fire-and-forget callers just call play. cancel_all is what CombatMaxVfxCleanupPresenter effectively does today — promoting it to the base contract lets the overlay tear everything down uniformly.

End state: the overlay is a dispatcher (~150 lines) that resolves effect_id → presenter via a registry map, applies reduced-motion/quality gates once (via the existing CombatVfxProfile / recipe-policy classes), and forwards. Adding a max-tier effect = new presenter file + one registry line (OCP), with zero edits to the overlay.

Apply the same audit to combat_vfx_presenter.gd (650 lines) when it next needs a change — same base contract.

Guardrails & Definition of Done

  • Size budget (baseline ratchet, enforced in review): record each inventory file's current line count as a checked-in baseline; a PR fails the ratchet only if it leaves a listed file above its baseline, and any PR that lands below the old baseline lowers it. This permits transitional facade delegators (the playbook's step 2 adds a few lines while removing many) while still forcing net shrinkage — judge the diff per-PR, not per-edit. New/extracted classes: ≤ 400 lines and ≤ 30 functions.
  • One extraction per PR. Behavior-preserving moves only; never mix a refactor with a feature or balance change.
  • Tests before moves. Every extracted service gets headless unit tests in scripts/tests/; the existing 2,825-line combat_model_test.gd should also be split alongside Phase 2 so tests mirror the new module boundaries.
  • No new autoload reach-ins. Extracted classes take dependencies via constructor/bind(). Grep gate: get_node("/root/ must not appear in any new service file.
  • Facade deletion is part of done. A strategy is complete only when the delegating methods are removed — a facade left forever is just a thinner god object.
Definition of done for the whole effort: no non-test script over 800 lines; RunState and CombatController contain wiring only; adding a shop offer type, settings row, or VFX effect requires no edits to any file on the inventory list.
