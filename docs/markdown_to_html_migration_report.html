Markdown to HTML Migration Report

The repository documentation set moved from plain-text pages into self-contained, browser-readable HTML: navigable, reviewable, shareable, and ready for richer internal wiki pages.

Executive Summary

The migration converted repository Markdown documentation, wiki pages, task trackers, asset briefs, QA reports, templates, addon docs, and repo-local memory notes into HTML counterparts. Original Markdown files were removed, and text references were updated to point at .html targets.

Runtime-facing run-log export was also changed from Markdown output to HTML output, so future generated run reports align with the new documentation format. Project instructions now state that future human-facing documentation should default to self-contained HTML.

This migration follows the same operating idea highlighted by Thariq Shihipar's HTML-output thread and companion examples: large agent-produced artifacts are easier to review when they can use layout, tables, diagrams, navigation, and browser-native interaction instead of staying as linear text.

Current status: migration is structurally complete in the working tree. Validation found no remaining Markdown-extension files and no stale Markdown-extension path references outside excluded Git/Android areas.

Source Article Takeaways

The captured article gives a practical rubric for when HTML is worth the extra generation cost. The migration report now treats those ideas as the rationale for Matchatro's documentation default, not just as a style preference.

Article Point Project Translation
HTML carries more kinds of information: tables, CSS design data, SVG, code snippets, interaction, workflows, spatial layout, and images. Use HTML for asset QA pages, architecture reports, design reviews, implementation plans, visual workflows, and status reports where layout improves comprehension.
Long Markdown documents are often not read deeply once they pass roughly 100 lines. Large Matchatro docs should bias toward navigable HTML with summary blocks, tables, status cards, and section links.
HTML can become a two-way artifact with sliders, knobs, local editors, and copy/export buttons. Future review tools such as asset approval boards, task triage pages, balancing dashboards, or prompt-tuning forms can be single-file HTML utilities.
HTML is easier to share because browsers render it directly once the file is available locally or hosted. Human-facing reports should be reviewable in the browser without build steps, external dependencies, or special Markdown preview tooling.
Tradeoffs are real: HTML is less token-efficient, takes longer to generate, and produces noisier version-control diffs. Keep Markdown-format exceptions available for short scratch notes, machine handoffs, strict external tooling, or changes where compact diffs matter more than reviewability.
A design-system reference helps avoid ugly or inconsistent generated HTML. Longer-term follow-up: create a reusable Matchatro documentation style/template so future HTML wiki pages share the same visual system.

Document Conversion

Each migrated page was emitted as a standalone HTML document with embedded CSS, semantic headings, a summary area, a generated table of contents, and a footer noting the source file. Original Markdown paths were removed and replaced by corresponding HTML paths.

Reference Rewrites

Repository text references were rewritten from Markdown targets to HTML targets. Asset metadata and manifest files were updated where they referenced converted briefs, reports, templates, or docs.

Run Log Export

Run-log reporting now uses an HTML builder and export facade. The old Markdown export functions were removed from the active runtime API surface and replaced with run_log_export_html().

Future Documentation Rule

The durable project rule now says that future wiki pages, product docs, architecture notes, task trackers, runbooks, decision records, asset reports, QA reports, and long-form review artifacts should default to self-contained HTML.

Risks and Follow-Ups

Agent discovery compatibility: the repo now has AGENTS.html instead of the conventional Markdown agent-guide filename. This matches the all-HTML migration request, but some external tools may still expect the conventional Markdown filename.
  • Some third-party tooling may assume README, CHANGELOG, LICENSE, or AGENTS files are Markdown. If that becomes a blocker, add a narrow exception or tool-specific adapter rather than reverting the whole documentation migration.
  • Large generated HTML diffs are less compact than Markdown diffs. Human review should happen in browser view; git diffs remain usable for targeted source changes.
  • Future documentation agents must avoid creating new Markdown files unless the target tool strictly requires Markdown format.
